## 主要な新機能と実践活用

### `/config key=value` — プロンプトから任意の設定をその場で変える（v2.1.181）

今回の目玉です。`/config` のあとに `key=value` を付けるだけで、**セッションを離れずにどの設定でもその場で**変更できるようになりました — しかも **interactive モード・`-p`（ヘッドレス）・Remote Control** のいずれでも動作します。

```bash
# 軽いタスクなら thinking をオフにしてトークンを節約
/config thinking=false

# ヘッドレス実行でもそのまま効く
claude -p "lint の結果だけ要約して"
```

これまでは設定を1つ変えるのにメニューを開くか `settings.json` を編集する必要がありましたが、今は**作業の途中で1行**でトグルできます。6/15 のクレジット分離以降、「このタスクは thinking まで要らない」というときに `/config thinking=false` ですぐ切り、重いタスクでは戻す、という**タスクごとのコスト調整**がぐっと軽くなりました。（正確な設定キー名は公式の settings ドキュメントを参照してください。）[GitHub v2.1.181](https://github.com/anthropics/claude-code/releases/tag/v2.1.181)

### thinking 中の接続切れも自動リトライ — 長い段落は行単位で（v2.1.181）

昨日（v2.1.179）の**「ストリーミング中に接続が切れても部分レスポンスを保持」**に続き、回復力がもう一段進みました。今回は **モデルが thinking（推論）している最中に API 接続が切れても自動でリトライ**します — 以前はこのケースで `Connection closed while thinking` となりターンごと落ちていました。加えて、**長い段落は最初の改行を待たず行単位でストリーミング**されるため、長く考える応答が画面により速く・安定して流れます。

リモート・SSH・不安定なネットワークで**長い推論ターンを最後の瞬間に失う**という失敗モードが、2日かけて両方向（部分レスポンス保持＋thinking リトライ）から塞がれた形です。自動化・CI で長いターンを回すなら、体感の安定性が上がります。[GitHub v2.1.181](https://github.com/anthropics/claude-code/releases/tag/v2.1.181)

## 開発者ワークフローティップス

### foreground サブエージェントもついに「5階層の深さ制限」に従う（v2.1.181）

今回の修正の1つは、実質的にコストのガードです — これまで **foreground サブエージェントはネストしたチェーンを無制限に起動**できました（5階層の深さ制限は *background* サブエージェントだけが対象でした）。v2.1.181 で **foreground も同じ5階層の上限**に従うようになりました。

実践的な意味は明確です。6/10 の「ネストしたサブエージェントは階層ごとにトークンが倍々で増える」、6/14 の「並列サブエージェントは3〜5個が現実的な上限」を思い出すと — **深さが無制限だった foreground 経路こそ、fan-out が暴走する穴**でした。サブエージェントがさらにサブエージェントを呼んで延々と下りていくケースが5階層で止まるので、クレジットが分離された今、**意図しないコスト暴走をランタイムが一度せき止めてくれる**わけです。深くネストするワークフローを組んでいるなら、この上限を前提に段階を設計し直しておきましょう。[GitHub v2.1.181](https://github.com/anthropics/claude-code/releases/tag/v2.1.181)

### タスクが変わったらコンテキストをクリアする — メッセージあたりトークン30〜50%削減

複数の実務ガイドが同じ結論に集まります — **長時間セッションには終わったタスクの会話履歴が古いまま溜まり**、各メッセージがその累積分までトークンとして運びます。タスクの切れ目で**コンテキストをクリアする（`/clear` など）と、メッセージあたりのトークンコストが最大30〜50%**減るという分析です。

同じガイドが対で強調するのが**「サブエージェントは1つにつき1役割」**という原則です — 何でもこなす "do-everything" エージェントは、description があらゆるものにマッチして、結局どれもうまく委譲されません。リサーチはサブエージェントに切り出してメインの会話をきれいに保ち、タスクが変わればコンテキストをクリアする — この2つの習慣が、**コストと精度を同時に**取りに行く最も安価なレバーです。上の `/config`・サブエージェント深さ制限と合わせ、「どこでトークンが漏れているか」を減らす今週の実践的なまとめです。[SmartScope — Claude Code 上級ベストプラクティス](https://smartscope.blog/en/generative-ai/claude/claude-code-best-practices-advanced-2026/)

## セキュリティ・制限事項

### Fable 回収の「本当の理由」が表面化 — 中国系アクセスの疑いと「直せば解除する」（6/13〜6/16）

6/12 の **Fable 5・Mythos 5 回収**の背景が、より具体的になりました。**Semafor の報道**によれば、ホワイトハウスを動かしたのは単なる jailbreak そのものではなく、**「中国系グループが上位モデル Mythos にアクセスしたという疑い」**でした — モデルが**リバースエンジニアリングや distillation で複製されうる懸念**です。トランプ政権の David Sacks（前 AI 担当）は X に政権側の説明を投稿し、**「テストパートナーが Fable のガードレールを回避して Mythos 級のサイバー能力に届く jailbreak を報告したが、Anthropic が修正を拒否したため、已むなく輸出規制を出した」**と主張しました。重要なのは彼が添えた **off-ramp（出口）**です — **「jailbreak がパッチされれば規制は解除する。ボールは Anthropic 側にある。」**

Anthropic は真っ向から反論します — この回避は**狭く普遍的でなく**（要は「コードベースを読んで欠陥を探せ」というレベル）、**OpenAI の GPT-5.5 など他の公開モデルでも同じ結果**が出る、というものです。中国アクセスの件についても広報は、**「ホワイトハウスは Fable の jailbreak をめぐる会話で中国の Mythos アクセスを持ち出していない。Anthropic は中国国内からの製品アクセスを遮断している」**と述べています。

開発者にとっての要点は2つです。第一に、輸出規制の動機が「抽象的な安全性」から**具体的な地政学（distillation・国家アクターのアクセス）**へと絞り込まれた点 — これは規制がより外科的になる一方で、より覆しにくくなりうることを意味します。第二に、それでも**「パッチ → 解除」という出口が公に示された**点 — 無期限の回収ではなく、条件付き復旧の道です。ただし韓国・日本などの外国籍開発者は**今も両モデルにアクセスできない**状態なので、実務対応は変わりません — `fallbackModel`・`availableModels` で Opus 4.8 などの代替経路を用意しておきましょう。[Semafor](https://www.semafor.com/article/06/13/2026/white-house-move-to-limit-anthropic-linked-to-concerns-about-chinese-access-to-mythos) · [Tom's Hardware](https://www.tomshardware.com/tech-industry/artificial-intelligence/trump-adviser-david-sacks-says-anthropic-refused-to-fix-fable-5-jailbreak-before-us-export-controls)

## エコシステム＆プラグイン

### MemoryWeft — 共有可能なグラフベース AI メモリ MCP（新規）

複数の AI・複数の環境のあいだで**メモリを共有・検索**できるようにするオープンソースの MCP サーバーです。AI との会話やドキュメントを **SQLite または PostgreSQL** に保存し、自宅↔オフィス、あるいはチームメンバー間で**作業コンテキストを持ち運べる**ようにします。

特徴は、**事前定義の関係スキーマが不要なイベント中心のナレッジグラフ**である点です — Graph+RAG で、データが溜まるほど関係が**継続的に育っていきます。**マネージャー UI とサンプル DB（ビートルズ Wikipedia・シャーロック・ホームズのテキスト）が同梱され、**セットアップなしですぐ**試せます。ここ数日扱ってきた「エージェントメモリ」の系譜を、MCP 標準の上で**複数のエージェントが一緒に使う共有記憶**として解いた試みです。[GitHub](https://github.com/rawdev/MemoryWeft)

## 知っておくと便利な小さな変更点

- **`CLAUDE_CLIENT_PRESENCE_FILE` 追加**：マーカーファイルを指しておくと、**自分がそのマシンの前にいる間モバイルのプッシュ通知を抑制**します（v2.1.181）
- **同梱 Bun ランタイムを 1.4 に更新** / **`sandbox.allowAppleEvents`** オプトイン（サンドボックスコマンドが macOS の Apple Events を送れる）（v2.1.181）
- **サブエージェントパネルの整理**：idle なサブエージェントは **30秒後に自動で隠れ**、リストは**最大5行**（スクロールヒント付き）、キーボードヒントをフッターに表示（v2.1.181）
- **フルスクリーンの URL を開くのに modifier が必要に**：フルスクリーンではリンクを **Cmd/Ctrl+クリック**で開くようになり、ネイティブターミナルの挙動に揃いました（v2.1.181）
- **起動時の性能リグレッション修正**：2.1.169 で入った**起動時 ~120ms の遅延**（MCP サーバー未設定時に managed-settings の fetch を待つ）と、ネットワークが遅いときに**空白ターミナルが最大15秒**止まる問題が修正されました（v2.1.181）
- **ネットワークドライブのファイル破損修正**：ネットワークドライブ・クラウド同期フォルダで Write/Edit が **0バイト・切り詰められたファイル**を作る問題が修正されました（v2.1.181）
- **`claude mcp get`/`list` のステータス表示の正確化**：tools/list が失敗しても `✓ Connected` と出ていたのを、**`! Connected · tools fetch failed`** とエラー付きで表示するようにしました（v2.1.181）
- ほかにも、**モデル切り替え直後に `/recap`・会話フォークが以前のモデルを使う問題**、**サブエージェントの「Thinking」時間が親の経過時間で表示される問題**、**AskUserQuestion のマルチセレクトで入力した「Other」回答が落ちる問題**などを修正（v2.1.181）

## おすすめコラム＆読み物

- **「エージェンティックなコードレビュー」（Addy Osmani）**：AI エージェントはコード産出を**4倍**にしても、実際の生産性向上は**約12%**にとどまる — その理由を、**ボトルネックが「作成」から「検証」へ移った**ことに求めます。モデルの推論はコードを吐き出すと消えてしまうため、レビュアーは**失われた意図を再構築**せねばならず、そこで生じる**「理解の負債（comprehension debt）」**がレビューをはるかに遅く・難しくする、というわけです。結論は鋭く — 2026年に先を行くチームは最も多くコードを生むチームではなく、**「テストが通る」と「これが何をするか理解している」を切り分けるレビュー体制**を持つチームだとします。6/17 の「select-and-justify なデザインワークフロー」、6/14 の「AI がエンジニアを置き換えなかった理由」と同じ場所 — 生成が安価になるほど、**信頼と検証**こそが仕事になります。[Addy Osmani](https://addyo.substack.com/p/agentic-code-review)
- **「来たる知能爆発に人類は備えができていない」（The Economist 寄稿）**：AI が**再帰的自己改善**に到達すれば、制御不能な「知能爆発」が数年内に引き起こされうるという警告です。著者は、AI のリーダーたちが私的には破滅的リスクを **10〜50%** と見積もる一方、**安全への投資は開発投資の約100分の1**にすぎない不均衡を指摘し、核軍縮に準じる**米中間の拘束力ある検証条約**が、自発的な措置では埋まらない隙間を埋めねばならないと主張します。6/15 の「Fable 事件が落とす巨大な影」、6/14 の Public Record 調査（規制支持70%超）と重ね — 能力が指数的に上がるとき、ガバナンスがどこまで追わねばならないかを正面から問います。[The Economist](https://www.economist.com/by-invitation/2026/06/15/humanity-isnt-ready-for-the-coming-intelligence-explosion)
- **「見知らぬ人にメールを送る理由」（Good Internet Magazine）**：拒絶の恐れがあっても、**見知らぬ人に思慮深いメールを送ること**が意味ある人間的なつながりだとする短いエッセイです。揮発性の高いソーシャルメディアと違い、メールは**永続的で、即時の返信を強要しない「人の速度」**で、より深く意図的なやり取りを可能にすると見ます。エージェントがコミュニケーションまで自動化する時代に、**遅く直接的なつながり**の価値を思い出させてくれます。[Good Internet Magazine](https://www.goodinternetmagazine.com/why-i-email-complete-strangers/)

## 注目プロジェクト＆ツール

- **sfs — 複数エージェントが1つのワークスペースを共有する同期ファイルシステム**：デバイス・セッションごとに**エージェントのコンテキストが別々に溜まる**問題を解くため、複数の AI エージェントが**共通のワークスペースを共有**できるよう同期するファイルシステムです。**S3・GCS と連携**し、誰が・いつ・どのデバイスでファイルを変えたかの**変更追跡ログ**を残し、Google Drive 式のストリーミングと違って**ファイルは常にローカルディスク**にあるため、エージェントが遅延なく即座に読めます。**オフラインファースト**で動作し、再接続時に自動同期、同時編集は**決定論的な競合解決**で整理します。任意のフォルダを同期ボリュームとしてマウントでき、**どのエディタ・エージェント**とも特別な統合なしに繋がります — 上の MemoryWeft が「共有記憶」を狙うなら、sfs は「共有ワークスペース」を狙う対です。[GitHub](https://github.com/runbear-io/sfs)
