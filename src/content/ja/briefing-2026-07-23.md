## 主要な新機能と実践活用

### `/deep-research`もopt-inに — 自動実行縮小の流れの三番目のスキル、`/code-review`は実行方式そのものが変わった (v2.1.218)

7/19(v2.1.215)で**`/verify`・`/code-review`をClaudeが自動実行しないよう**変更されたのに続き、v2.1.218は**`/deep-research`も同じ流れに移しました** — Claudeはもう自分の判断で`/deep-research`を開始せず、**ユーザーが明示的に呼び出したときだけ**動きます。

- **`/deep-research`のopt-in化**: リサーチスキルはウェブ検索を扇状に展開する重い処理です — 意図しないタイミングで自動起動しトークンと時間を消費していた問題が、呼び出し主体を人間に固定したことで解消されます。
- **`/code-review`は実行方式が変わる**: opt-in化にとどまらず、**`/code-review`が今回からバックグラウンドサブエージェントとして実行**されます — レビュー作業がメインの会話画面を占有しなくなり、**積み重なったスラッシュコマンドそのものをレビュー対象として維持**します。レビューを走らせながら、同時に別の作業を続けられるということです。
- **`/ultrareview`・`/code-review ultra`も併せて改善**: `/ultrareview`に「review my auth changes」のような文章形式の引数を渡すと失敗していた問題が修正され、今後は**現在のブランチをレビューしつつそのテキストをノートとして添付**します。`/code-review ultra`が非対話型セッションで黙ってローカルレビューに切り替わっていた問題も修正され、**明示的にクラウドレビューを実行**します。

```text
/deep-research   # もう自動起動しない — 必要なときに直接呼び出す
/code-review     # バックグラウンドサブエージェントとして実行 — 会話を止めない
/ultrareview review my auth changes   # 失敗せず現在のブランチをレビュー + ノート添付
```

ポイントは**「opt-in化がひとつのスキルにとどまっていない」**という点です — 7/19の`/verify`・`/code-review`から今日の`/deep-research`まで、重いスキルをClaudeが自動的に起動していた慣性が、リリースのたびに一枚ずつ剥がされています。無人ループでこれらのスキルの自動実行に何となく頼っていたなら、CLAUDE.mdやhookで呼び出しルールを明文化し直すタイミングです(7/20ブリーフィングのチェックリスト参照)。[GitHub v2.1.218](https://github.com/anthropics/claude-code/releases/tag/v2.1.218)

### auto modeの判断委譲がさらに一段広がった — dangerous-rm・バックグラウンド`&`・plan modeの証明不能なBashまでclassifierが担う (v2.1.218)

v2.1.218は**auto modeで権限ダイアログの代わりにauto-mode classifierが直接判断する範囲**を広げました。

- **危険コマンド検査の委譲**: **dangerous-rm(危険な削除コマンド)・background-`&`(バックグラウンド実行)・suspicious-Windows-path(怪しいWindowsパス)**の検査は、もう権限ダイアログを開かず、**auto-mode classifierが調整**します。
- **plan modeも同じ方向**: **静的解析では読み取り専用だと証明できないBashコマンド**について、plan mode with autoはもうプロンプトを出さず、**classifierが代わりに判断**します。

ポイントは**「静的ルールで拾いきれないグレーゾーンの判定権がclassifierに移りつつある」**という点です — auto mode・plan modeの摩擦が減る一方、その判断への信頼をclassifierに委ねるトレードオフでもあります。auto modeを無人パイプラインで使っているチームは、今日から上記4カテゴリのコマンドがプロンプトなしで通過するかどうか、一度観察しておくと安全です。[GitHub v2.1.218](https://github.com/anthropics/claude-code/releases/tag/v2.1.218)

## 開発者ワークフローティップス

### `context: fork`スキル、デフォルトでバックグラウンド実行されるように (v2.1.218)

v2.1.218は**`context: fork`で宣言したスキルをデフォルトでバックグラウンド実行**するよう変更しました — 個々のスキルで`background: false`を指定すればオプトアウトできます。

- **何が変わるか**: これまで`context: fork`スキルをフォアグラウンドで実行していたワークフローがあった場合、アップグレード後は**デフォルト動作がバックグラウンドで動く方に変わります** — スキルの結果を即座に会話へ反映させたい構成であれば、`background: false`を明示しないと以前の挙動は維持されません。
- **今日のチェック項目**: `.claude/skills/`に`context: fork`を使っているスキルがあれば、アップグレード後に**意図どおりフォアグラウンド/バックグラウンドが分かれているか**を一度確認しましょう。

```yaml
# SKILL.md frontmatter — これまでどおりフォアグラウンドを維持するには
context: fork
background: false
```

### Trustダイアログ、承認対象リポジトリのルートを明示するように (v2.1.218)

v2.1.218は**trustダイアログが承認対象リポジトリのルートパスを明示**するよう改善しました — 信頼を付与する際、*正確にどこまで承認しているのか*をダイアログ自体で確認できます。worktreeやサブフォルダを行き来しながら作業するチームは、信頼を承認するたびにこのパス表示を習慣的にチェックしておくと安全です。[GitHub v2.1.218](https://github.com/anthropics/claude-code/releases/tag/v2.1.218)

### Gitで`--end-of-options`が必要な理由 — 信頼できないrevisionを安全に扱う方法

Gitの`--`はよく知られる一般的なオプション終端子ではなく、**リビジョンとパス指定(pathspec)を区切るセパレータ**である点を指摘する実践的なTipsです。ユーザー入力や外部データに由来する**信頼できないrevision文字列**を`git log`・`git diff`のようなコマンドに安全に渡すには、Git 2.24.0から対応している**`--end-of-options`**が必要です。

```bash
# revisionが-で始まるオプションとして解釈されるリスクを遮断
git log --end-of-options "$rev" -- "$path"
```

- **なぜ重要か**: エージェントが自らgitコマンドを組み立てて実行するワークフロー(自動コミットログ照会、PR diff解析、revisionベースのスクリプト)が増えるほど、**外部由来の文字列をそのままgitコマンドに埋め込むコード**はインジェクションに近いリスクを抱えます。`--end-of-options`はそのリスクを取り除く標準的でシンプルな防御線です。

ポイントは今週ずっと続いている**ハードニング連鎖**(worktree・シンボリックリンク・権限チェッカーのパース不整合)と同じ地点にあります — Claude Code自体ではなく、**Claude Codeが代わりに組み立てて実行するgitコマンド**を書く開発者にも同じ境界線が当てはまります。エージェントにgitを自動化させるスクリプトやhookを書いているチームなら、今日チェックする価値があります。[GeekNews](https://news.hada.io/topic?id=31698)

## セキュリティ・制限事項

### ハードニング連鎖の七番目の輪 — agent frontmatter hookの未信頼フォルダでの実行を遮断、静かな信頼性不具合もまとめて解消 (v2.1.218)

v2.1.218には7/15から続く**信頼境界のハードニング**の七番目の輪が収録されました。今回の標的は*エージェント定義ファイルそのものの信頼境界*です。

- **agent frontmatter hookの遮断**: **エージェントのmarkdownファイルにあるfrontmatter hookが未信頼(untrusted)フォルダで実行されていた問題**が修正されました — 今後、**hookはそのエージェントファイルが置かれたフォルダ自体がワークスペースの信頼を受け入れている場合にのみ**実行されます。7/20(worktree・シンボリックリンク)、7/22(バックグラウンドセッション隔離)に続く、*信頼境界がファイルの所在ひとつでさらに絞り込まれていく*延長線上にあります。
- **エージェント名の予約語保護**: **`:`を含むエージェント名は拒否**されます — `:`はプラグインの名前空間専用として予約されています。
- **静かな信頼性不具合もまとめて解消**: このほかにも、**深くネストされたwatchディレクトリツリーが削除・移動される際、および深くネストされたUIツリーをレンダリングする際に発生していたスタックオーバーフロークラッシュ**、**圧縮(compaction)後にfork-sessionの系譜が失われる問題**、**破損したdelta添付を持つトランスクリプトでレジュームしたセッションが毎ターン失敗していた問題**、**リモートセッションがワーカー交代後もハートビートを送り続け、デスクトップ・IDEプロセスが拒否されたリクエストを数秒おきに永遠にリトライしていた問題**まで、一緒に解消されました。

ポイントは**「信頼境界のハードニングと長時間セッションの信頼性修正が、同じリリースの中で毎週繰り返されるパターンになった」**という点です — カスタムエージェントやhookを幅広く使っている、あるいは`/fork`・リモートセッションを常時回しているチームは、v2.1.218にアップグレードした上で両系統の前提を併せて再確認してください。[GitHub v2.1.218](https://github.com/anthropics/claude-code/releases/tag/v2.1.218)

### 7/22のインシデント4件 — Sonnet 5・Haiku 4.5のelevated errorsを含め、すべて解消 (7/22)

StatusGatorの記録によると、**7/22には4件のインシデント**が発生しました — **Sonnet 5のelevated error rates(午後3:28、45分間)**、**サービス中断(午後5:28、20分間)**、**Microsoft Officeアドインの可用性問題(午後5:58、5時間55分間)**、**Haiku 4.5のelevated errors(午後11:53、20分間)**です。**いずれも解消済み**です。v2.1.218が公開された時間帯と重なるため、昨日おかしな挙動に遭遇した場合は、バージョンの問題なのかインシデントの時間帯なのかをまず切り分けてください。[Claude Status](https://status.claude.com/history) · [StatusGator](https://statusgator.com/services/claude)

## エコシステム＆プラグイン

### OmniRoute — 散らばった無料・低価格AIティアをひとつのローカルゲートウェイに束ねる

無料・低価格のAIアクセス手段がプロバイダーごとに散らばっている問題を狙ったローカルゲートウェイのプロジェクトです。**ひとつのローカルエンドポイント(`localhost:20128/v1`)で271のプロバイダー・500以上のモデルを接続**し、**Claude Code・Codex・Cursor・Cline・Copilotなど26のコーディングツールをひとつの設定で統合**して使えます。

- **何を束ねるか**: **90以上の無料ティアと40以上の永久無料アカウント**を統合し、月間約14億トークン規模の無料アクセスをひとつのゲートウェイの背後に集約します。
- **どこで使うか**: 複数のコーディングツールを行き来しながら無料・低価格モデルを試している開発者にとって、**ツールごとにAPIキーや設定を個別に管理していた手間**をゲートウェイひとつで代替できる選択肢です。

ポイントは今週のブリーフィングで繰り返し取り上げてきた*モデルのコモディティ化*(7/18 State of Open Source AI)の実物版ツールだという点です — モデルの選択肢が増えるほど、その選択肢を**一箇所でルーティングする層**に価値が生まれるというシグナルです。[GeekNews](https://news.hada.io/topic?id=31710)
