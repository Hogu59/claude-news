## 主要な新機能と実践活用

### Claude for Teachers — Claude Code・Coworkを米国K-12教師に無料で (7/14)

Anthropicが**認証済みの米国K-12教師にプレミアムClaudeを無料提供するClaude for Teachers**をローンチしました(7/14)。開発者向けの発表ではありませんが、**エージェント自動化が開発者を越えて非技術職へ広がる**今週最大の実例なので、ここに置きます。

- **何が無料か**: 多くの教師が無料プランでは目にしない**有料ティアのツール — Claude CodeとClaude Coworkをまるごと**含みます。さらに**全50州の学業標準を担うLearning Commonsコネクタ**と、**学習科学者と共に作った教授(teaching)スキルセット**が付きます。
- **Claude Code的な角度 — 委任型の自動化**: Coworkのスケジュール自動化で、*その日のexit ticketをレビューして明日の授業を調整する*という仕事を**一度任せれば、登校日ごとの午後4時に自ら実行**します。Claude Code・Coworkがセッションを越えて仕事を引き継ぐ、あの委任パターンが今回は教師の反復業務に載ります。
- **データとガバナンス**: 教師が共有したものは**モデルの学習に使われず**、規約は**米国教師連盟(AFT)と共同開発したFERPA準拠**の形です。**2027/6/30までに登録すれば1年間無料。**

要点は、**Claude Codeが開発ツールを越えて「専門職への委任インフラ」として梱包・配布され始めた**ことです。7/12のClaude for Government(FedRAMP High)、7/13のM365書き込みコネクタが*規制・行動チャネル*を広げたのに対し、今回は*非開発職そのもの*を狙います。開発者に直接は届きませんが、**同じエージェントハーネス(スキル・コネクタ・スケジュール自動化)が、ドメインスキルとして梱包されるとどこまで広がるか**を示す座標です。 [Anthropic](https://www.anthropic.com/news/claude-for-teachers) · [9to5Mac](https://9to5mac.com/2026/07/14/anthropic-is-giving-teachers-free-access-to-premium-claude-features-details-here/)

### 7/14の三連リリース — スクリーンリーダーモードと「静かなパフォーマンス大改善」 (v2.1.208)

3つのうち中身があるのはv2.1.208です。目立つ新機能と、目立たないが体感の大きいパフォーマンス改善が一緒に入っています。

- **スクリーンリーダーモード(アクセシビリティ)**: 視覚障害のあるユーザー向けに**プレーンテキストレンダリング**をopt-inで提供します — `claude --ax-screen-reader`、`CLAUDE_AX_SCREEN_READER=1`、または設定に `"axScreenReader": true`。TUIのリッチな描画がスクリーンリーダーと衝突していた問題に正面から取り組んだ、最初のアクセシビリティ機能です。
- **「静かなパフォーマンス大改善」**: **ツール呼び出しあたりのCPUオーバーヘッドを高ツール状況で約7倍**、**編集の多いセッションのトランスクリプトサイズを最大79倍**削減し、**複数のメモリリーク(MCP stderr蓄積・LSPドキュメント・async hook)**を修正、大きなファイル編集キャッシュを16MBに上限設定しました。200行を超える大きなmarkdownテーブルで固まる問題、極端に長い1行のファイルで読み込みが遅くなる問題も解消されています。
- **細かな便利さ**: vimモードの**2文字シーケンスリマップ**(`vimInsertModeRemaps`、例: `jj`→Escape)、フルスクリーンのマルチセレクトと「Other」入力行の**マウスクリック**対応、企業配布用ランチャーをagent view・バックグラウンドサービスに注入する `CLAUDE_CODE_PROCESS_WRAPPER` が付きました。

要点は、**今回のリリースの本当のニュースは新しいボタンではなく「実行コスト」**だということです。下のワークフローティップで扱うとおり、長時間エージェントセッションや大規模MCPスタックを回すチームにとって、このパフォーマンスバッチは*機能アップデートではなくインフラアップデート*です。 [GitHub v2.1.208](https://github.com/anthropics/claude-code/releases/tag/v2.1.208)

## 開発者ワークフローティップス

### 長時間・エージェント・大規模MCPセッションなら今すぐアップグレードを — 今回は「パフォーマンスが機能」 (v2.1.208)

最近のブリーフィングが繰り返してきた命題が*コンテキストに載るものすべてが注意力の予算を使う*(7/9・7/12)なら、v2.1.208はその予算の**CPU・メモリ軸**を正面から改善しました。無人・並列・長時間のセッションを回すなら、そのままチェックリストです。

- **どのセッションが最も得か**: **ツール呼び出しの多いセッション**(大規模MCPスタック・多数のサブエージェント)は、ツール呼び出しあたりのCPUが約7倍下がるので体感が大きいです。**編集の多い長時間セッション**は、トランスクリプトが最大79倍小さくなり、再開(resume)とコンテキスト読み込みが速くなります。
- **原因不明の「遅くなる・固まる」を経験したなら**: MCP stderr蓄積・LSPドキュメント・async hookのリークが今回修正されました — 数日回る無人セッションが徐々に遅くなっていた原因はここかもしれません。上げてから再現するか確認してください。

```bash
# アップグレード後、長時間セッションの体感を確認
claude update            # またはパッケージマネージャで v2.1.208 以上に
/doctor                  # インストール・コンテキスト予算の点検(7/10・7/12のティップと併せて)
/context                 # 重いセッションのトークン・ツール使用を監査
```

パフォーマンス改善は静かなので見逃しやすいですが、長時間エージェント運用では**最も直接的なコスト削減**です。新機能リストが薄く見えるからといってこのリリースを飛ばさないでください — このバッチの恩恵を受けるのは、*最も長く、最も大きく*回すまさにそのチームです。 [GitHub v2.1.208](https://github.com/anthropics/claude-code/releases/tag/v2.1.208)

### 並列worktreeエージェントのワークフローをv2.1.210でハードニングする

複数のエージェントを隔離されたgit worktreeで並列に回すパターン(7/13・7/14のサブエージェント・worktreeの流れ)が広まる中で、v2.1.210はそのパターンの**鋭い角を2か所**削りました。

- **worktree隔離が実際に隔離されるように**: `isolation:'worktree'` サブエージェントが**メインリポジトリにgit変更コマンドを加えていた**問題が修正されました — 並列worktreeで衝突を避けるための、まさにその安全装置が漏れていた穴を塞ぎました。並列エージェントが互いのブランチを踏んでいた原因不明の事故があったなら、ここが原因かもしれません。
- **誤設定の権限ルールを起動時に捕まえる**: `Write(path)`・`NotebookEdit(path)`・`Glob(path)` の形の権限ルールに**起動時警告**が付きました — ルールを誤って書いて意図したファイルが保護されない状況を黙って見過ごさず、起動時点で知らせてくれます。
- **コストの角度もひとつ**: auto modeの**権限分類器のデフォルトがSonnet 5**になりました — 無人セッションで各アクションを判定するこの分類器のモデルが変わったので、auto modeパイプラインのコストと判定傾向を一度点検してください。

```json
// .claude/settings.json — ルールを明示的に置き、起動時警告で検証する
{
  "permissions": {
    "deny": ["Write(./.github/**)", "Write(./infra/**)"],
    "ask":  ["Glob(./secrets/**)"]
  }
}
```

要点は7/11の*デフォルトが便利な側へ動いたので、安全は明示的に守れ*と同じ位置です — 今回はCLIが起動時点で**並列実行の隔離と権限ルールの整合性**を検証してくれます。 [GitHub v2.1.210](https://github.com/anthropics/claude-code/releases/tag/v2.1.210)

## セキュリティ・制限事項

### v2.1.210、injectionと破壊的コマンドの境界を締めるハードニング (7/14)

v2.1.210は、目立つ便利機能と同じくらい、**エージェントを狙う攻撃面**を狭める修正を一緒に含んでいます。無人・自動トリガー・サブエージェントを使うチームなら、見過ごせない項目です。

- **Agentツールのprompt injectionハードニング**: **サブエージェントが読んだ内容経由のprompt injection**に対してAgentツールを強化しました — エージェントが外部ファイルやweb内容を読み、上位の実行フローを操られる経路(7/9のGitLostが示した類型)をもう一枚塞ぎました。
- **ultracode opt-inの非人間入力での発火を遮断**: 超高強度モードを有効にする `ultracode` キーワードのopt-inが、**webhookやPRコメントのような人間でない入力**で発火していた問題が修正されました — 外部から流れ込んだテキストが実行強度を勝手に引き上げる**実際のinjectionベクター**を封じたものです。自動トリガーでClaude Codeを繋いだパイプラインなら重要な修正です。
- **破壊的削除に確認を強制**: `$(…)`・バッククォートコマンド内の**破局的(catastrophic)削除**が、今や**`--dangerously-skip-permissions` 下でも確認を要求**します(v2.1.208) — 「権限スキップ」ですら完全に無防備ではなくした、無人セッションの最後の安全ピンです。

要点は、最近のブリーフィングが追ってきた**「便利さのために広げた信頼境界を、一枚ずつ再び狭める」**(7/8 SkillCloak、7/9 GitLost、7/11 v2.1.207ハードニング)の今週の延長です — 特に*自動トリガーとサブエージェントの内容*が実行を操る経路が今回の標的です。サードパーティのトリガーでエージェントを繋いでいるなら、v2.1.210に上げ、自動化の入力境界を点検し直してください。 [GitHub v2.1.210](https://github.com/anthropics/claude-code/releases/tag/v2.1.210)

## おすすめコラム＆読み物

- **「2026年になぜコードを書くのか」**: *AIがコードの大半を生成する時代に、エンジニアがなお自分でコードを書くべき理由*を突いた記事です(softwaredoug、7/9)。著者自身*コードの圧倒的多数はAI生成だ*と認めつつ、直接コーディングの価値は**英語(自然言語)ではなく実行環境で思考すること**にあると見ます — 核心の一文は*英語は計算を正確に表現するほど精密ではない*で、コードの実行はコードレビューだけでは見えない問題を露わにします。LLMエージェントを*コンパイラではなくジュニアインターン*にたとえ、既存の決定を保守的に温存し、防御的なレイヤーで人間の間違いを増幅すると指摘し、**反復を委任する前に、まず自分の実験でパターンを立てよ**と勧めます。昨日(7/14)のブリーフィングの*外部ループを所有せよ*が**判定**の所有を語ったなら、この記事はその手前 — **理解**の所有がどこから来るかを語ります: 他人の作ったdiffを受動的にレビューするだけでは、システム理解は育たない、と。エージェントへの委任を増やすチームほど、*どこまで自分の手を入れるか*の基準線として読む価値があります。 [softwaredoug](https://softwaredoug.com/blog/2026/07/09/write-code)
- **「ジュニアはタスク消化のために雇うのではない」**: **なぜジュニアエンジニアを雇うのか**についてのKent Beckの記事です。会社は*即時の生産性*ではなく*組織の能力を引き上げる潜在力*を見てジュニアを雇い、成功は**消化したチケット数ではなく、信頼性(動くコード・明快なコミュニケーション・合理的なスケジュール)と学習速度**で測られる、というものです。著者はジュニアを3つに分けます — *不要な仕事をなくしチームの生産性を増幅する転換型(A)*、*着実に成長する信頼型(B)*、*1年以内に去る可能性が高い層(C)* — そして真の卓越は**「90%の利益を生む10%の仕事」を見極める**より深い活動から来ると言います。AIがタスク完遂そのものを安くする今、この記事はまさに反対側を狙います — *スループットが指標になった瞬間、人間もエージェントのように評価される*という罠への警告であり、上の「なぜコードを書くのか」と対をなす、**人間の役割**の再定義です。 [Kent Beck](https://newsletter.kentbeck.com/p/hey-n00b-we-didnt-hire-you-to-complete)

## 注目プロジェクト＆ツール

- **Brain-AI Memory — 長時間エージェントの「記憶失敗」を類型学で診断**: 長く回るLLMエージェントの記憶失敗を*単純な検索(retrieval)問題*で一括りにせず、**エピソード/意味記憶・手続き規則・数値状態・ルーティング・入力ゲート**に分け、それぞれの失敗条件とライフサイクルを定義したオープンソースアーキテクチャです。作者は複数プロジェクトのマルチエージェントシステムで数か月実測してから公開し、**60秒で実行できる例・再利用テンプレート・ベンチマーク**を併せて出しました — 特に*96個のキーワードポインタでインデックステキストを93%削減したが、recall@3が86.1%→71.0%に下がった*という**否定的な結果まで透明に**公開している点が目を引きます(フィードバック後、pipインストール・CLI・Command Centerを含むv0.2.0を公開)。Claude Codeで長時間エージェント・サブエージェントを回し、*コンテキストが長くなるにつれ何を忘れるか*を経験したチームに、**記憶を「類型別の失敗」として診断するチェックリスト**としてすぐ使えます。 [GitHub](https://github.com/Hahyun-Lee/brain-ai-memory)
- **QAMap — コミットとdiffを読んでPRのQAシナリオを提案するローカルCLI**: PRのボトルネックが*テストを書くこと*より**各変更が何を証明すべきかを決めること**にある、という問題意識から出発したローカルCLIです。ドメイン全体を理解しようとせず、**gitのコミットとdiffから変更の意図を読み**、正常系・失敗・エッジ・状態遷移の**QAシナリオを根拠と確信度とともに**提案します — すべてローカルで動きます(v0.4.0、実際の失敗事例で反復改善中)。上の「2026年になぜコードを書くのか」が突いた*diffを手作業でレビューするだけでは理解は育たない*の実務的な補完として読めます — **エージェントが作った変更のQA面を人間が見落とさないよう**図式化してくれる小さなツールです。 [GitHub](https://github.com/IvoryCanvas/qamap)
