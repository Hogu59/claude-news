## 主要な新機能と実践活用

### スタック型スラッシュスキル — スキルを『組み合わせて』1行で呼ぶ (v2.1.199)

v2.1.199 から、**プロンプトの先頭にスラッシュスキルを複数積むと、先頭のスキルを最大5個まで一度にロード**します。これまでスキルは1つずつ呼ぶものでしたが、*ワークフローの組み合わせ*が可能になりました。

```bash
# スキル A と B を同時にロードしてから指示 — 最大5個まで
/code-review /verify この diff をレビューして、実際に動くか検証して

# 例: 可視化ルール + フロントエンドルールを一度に適用
/dataviz /frontend-design ダッシュボード画面を作って
```

要点は**「スキルが単発コマンドから組み合わせ可能なビルディングブロックへ」**変わったことです — これまで「レビューしてから検証」のように2つのプレイブックを併用するには、スキルに別スキルの内容をコピペするか、順番に2回呼ぶしかありませんでした。6/29 の **CLAUDE.md・skill・hook の置き場所の区別**(skill は必要時のみロードされるドメインワークフロー)と合わせて読むと — 狭く鋭いスキルを複数用意し、**タスクごとに組み合わせてロードする**設計が公式にサポートされたことになります。チームのスキルを設計するなら、*何でもやる大きなスキル1つ*より*組み合わせ可能な小さなスキル群*に分割する理由がまた1つ増えました。 [GitHub v2.1.199](https://github.com/anthropics/claude-code/releases/tag/v2.1.199)

### デフォルト権限モードが「Manual」に — 自動進行がオプトインになった (v2.1.200)

v2.1.200 は**デフォルトの権限モードを「default」から「Manual」に**変更しました — CLI・VS Code・JetBrains 全体に適用されます。あわせて **`AskUserQuestion` ダイアログがデフォルトでは自動進行しなくなりました** — これまでは席を外すと一定時間後にダイアログが勝手に進みましたが、いまは**明示的に答えるまで待つのが既定**で、従来の idle timeout による自動進行は `/config` でのオプトインが必要です。

```text
# v2.1.200 以降のデフォルト動作
1. 権限モード: Manual — ツール実行前の確認が明示的な既定値
2. AskUserQuestion: 回答まで待機 — 自動進行は /config でオプトイン
```

重要なのは**「静かに勝手に進んでいた挙動が、1つずつオプトインに変わっている」**という方向性です — 6/29 の `--dangerously-skip-permissions` の静かなフォールバック修正、6/27 の外部プラグイン導入への同意強制と、まったく同じ流れです。実務的な含意も明確です: **旧来の半自動フローを前提にしたセッション習慣やチームのオンボーディング文書があれば、既定値が変わったので確認**してください。特に新しいマシン・新メンバーの初回セッションは以前より確認プロンプトが増えます — 自動化が必要な箇所は権限モードを明示的に設定し、人が見ているセッションでは新しい既定値のほうが安全です。 [GitHub v2.1.200](https://github.com/anthropics/claude-code/releases/tag/v2.1.200)

## 開発者ワークフローティップス

### 無人パイプラインのリトライ構成が一新された — 429 自動リトライ + ウォッチドッグ + 部分保存

v2.1.199 は無人実行の**失敗処理を3つ同時に**変えました。CI・cron・background で Claude Code を回しているなら、そのままチェックリストです。

- **429 の自動リトライ**: 一時的なサーバー rate-limit(429)が、**サブスクライバーには backoff 付きで自動リトライ**されます — 瞬間的な上限エラーで夜間パイプラインが死ぬ失敗モードが、既定で消えます。
- **リトライ・ウォッチドッグ**: `CLAUDE_CODE_RETRY_WATCHDOG` を有効にすると、**非容量系(non-capacity)エラーの既定リトライ回数が300に**上がります — 長い無人ジョブが一時エラーで早々に諦めないためのダイヤルです。
- **部分出力の保存**: ストリーミング応答の途中で overloaded・サーバーエラーが起きても、**そこまでの部分出力を破棄せず保存**し、不完全応答の通知を付けます — 長い生成が90%地点で丸ごと蒸発する問題がなくなります。

```bash
# 長い無人ジョブのリトライ耐性を上げる組み合わせ (v2.1.199+)
export CLAUDE_CODE_RETRY_WATCHDOG=1   # 非容量系エラーのリトライを既定300回に
claude -p "マイグレーションを実行してテストを通して"
# おまけ: SSL 証明書エラー(TLS 検査プロキシ・期限切れ証明書)はリトライを浪費せず
#   即時失敗 + 修正ヒントを出すようになり、企業プロキシ環境での診断が速くなる
```

要点は**「リトライすべきものは最後までリトライし、リトライしても無駄なもの(SSL 設定ミス)は即時失敗させる」**という分離です — idle watchdog の既定 ON(6/30)、ECONNRESET backoff(7/2)に続く、無人実行の信頼性シリーズの次章です。外側に自前のリトライループを巻いている場合は、**二重リトライで最悪待ち時間が膨らまないか**を一度点検してください。 [GitHub v2.1.199](https://github.com/anthropics/claude-code/releases/tag/v2.1.199)

### サブエージェントの『静かな失敗』が消えた — 『空の結果=成功』の仮定を捨てよ

2つ目も v2.1.199・200 由来ですが、含意が大きいので独立して扱います。サブエージェントの失敗処理が大幅に修正されました — ① rate limit・サーバーエラーで切られたサブエージェントが**静かに失敗する代わりに部分結果を返し**、② **API エラーを成功した結果のように報告するバグ**が修正され、③ v2.1.200 では rate limit で切られた場合に**空の結果ではなくクリーンな失敗**を返します。

これまでサブエージェント基盤のパイプラインには見えない穴があったわけです — *サブエージェントが空文字列を返したとき、成功か失敗か区別できず、API エラーが成功に化けることさえあった*。7/2 のブリーフィングで *Explore・サブエージェントにもっと重い仕事を任せよう*と勧めましたが、その委任が信頼に足るためにはこれらの修正が必要でした。

実務の処方箋は2つ。① サブエージェントの結果を消費する自動化があるなら、**「空の結果を成功とみなす」防御コードを外し、部分結果と失敗を区別して処理**するよう更新してください。② 今日のおすすめコラム『AI との冒険』が指摘するとおり — インフラの静かな失敗は直っても、**モデルの虚偽の完了報告(false completion claims)は残っています**。インフラ層はリリースが直してくれますが、モデル層は依然として**検証ハーネス(テスト・静的解析・機械的証拠)**が人間の仕事です。 [GitHub v2.1.199](https://github.com/anthropics/claude-code/releases/tag/v2.1.199) · [GitHub v2.1.200](https://github.com/anthropics/claude-code/releases/tag/v2.1.200)

## セキュリティ・制限事項

### Fable 5 セーフガードの公式詳細 + Cyber Jailbreak Severity(CJS)フレームワーク草案 (7/2)

昨日扱った「ガードレールがより厳しくなって戻ってきた」の**公式解説版**です。7/2、Anthropic が **Fable 5 のサイバー・セーフガードの詳細と jailbreak 深刻度フレームワークの草案**を公開しました。

- **4群の分類**: サイバー活動を**禁止(prohibited)・高リスク dual-use・低リスク dual-use・無害(benign)**の4グループに分けます — 防御的・一般的な IT 作業は許可しつつ、危険な用途を止める構造です。
- **分類器と fallback の正体**: Fable 5 の分類器は**報告された jailbreak の再現試行の99%以上をブロック**するよう配備され、**サイバーセキュリティ・生物/化学・distillation** 関連のリクエストを検知すると**応答を Opus 4.8 が代行**します — 昨日ワークフローのヒントとして観察ベースで扱った*静かな Opus ルーティング*の公式確認です。
- **CJS-0〜4**: 提案された **Cyber Jailbreak Severity スケール**は、jailbreak を**能力増加(capability gain)・増加の範囲(breadth)・武器化の容易性・発見の容易性**の4軸で採点・合算し、**CJS-1(Low、1〜3.5)・CJS-2(Medium、4〜6.5)・CJS-3(High、7〜8.5)・CJS-4(Critical、9〜10)**のバンドにマッピングします。
- **業界標準化**: Amazon・Microsoft・Google ら **Glasswing パートナー**とともに業界共通フレームワークとして磨いている段階です — 新しい jailbreak 発見のトリアージと、政府・業界への一貫したリスク伝達が目標です。

要点は**「脆弱性における CVSS のように、jailbreak にも共通の深刻度言語を作ろう」**という試みだということです — 7/2 のブリーフィングで扱った復帰条件(jailbreak 情報の共有・業界セキュリティ標準)が文書として具体化した最初の成果物でもあります。セキュリティ領域で Claude を使うチームは、自分たちのワークロードが4群のどこに落ちるか、fallback 発生時の挙動をこの文書基準で整理しておきましょう。 [Anthropic 発表](https://www.anthropic.com/news/fable-safeguards-jailbreak-framework) · [IT Brief](https://itbrief.asia/story/anthropic-widens-fable-5-safeguards-rates-jailbreaks)

### 7/3 夜の Fable 5・Mythos 5 elevated errors — 復帰後初のインシデント (7/3)

**7/3 21:54〜22:06 UTC(約12分)**、**Claude Fable 5 と Claude Mythos 5 の elevated errors** インシデントが発生しました — すぐに解消しましたが、**7/1 の復帰以降 Fable 5 として初のインシデント**である点は記録しておく価値があります。7/2 の Opus 4.8 elevated errors(約41分)に続き、7月も*短く反復する安定性ノイズ*のパターンが続いています。昨日の処方箋どおりです — Fable 5 がブロックされたときに fallback を受けるモデル(Opus 4.8)と Fable 5 自体が**同時に揺れうる時間帯**が存在するため、クリティカルな自動化にはリトライ・バックオフ(上のワークフローの v2.1.199 構成がまさにここを助けます)と失敗アラートを維持してください。 [Claude Status — Incident History](https://status.claude.com/history)

### ホワイトハウス、OpenAI・Google・Anthropic と『自主的リリース標準』を最終調整中 — 1週間以内に発表の見込み

Fable 5 騒動の**制度化フォローアップ**です。報道によると、ホワイトハウスは **OpenAI・Google・Anthropic と自主的(voluntary)な AI リリース標準**を最終調整しており、**早ければ1週間以内**に発表される可能性があります。骨子は**サイバーセキュリティ・ベンチマークとレビュー・タイムライン**を明文化し、Fable 5 の出荷を3週間凍結させたような**場当たり的(ad-hoc)な政府審査を置き換える**ことです。

開発者視点の含意は — 6/12〜6/30 の輸出規制凍結が*一度きりの事件*ではなく、**再現可能な手続きとして制度化**されつつあるという点です。予見可能性は上がりますが(レビュー・タイムラインの明文化)、*フロンティアモデルの出荷が政府ベンチマークを通過するのが標準になる時代*が固まるシグナルでもあります。上の CJS フレームワーク(業界側の準備物)とちょうど噛み合う構図です — モデルの可用性が政策カレンダーに左右されうるという前提は維持してください。 [AIToolsRecap](https://aitoolsrecap.com/Blog/ai-news-july-3-2026)

## おすすめコラム＆読み物

- **『コードレビューの主な目的は保守不能なコードを見つけること』**: コードレビューの目的を再定義する、短く鋭い文章です。核心は — レビューの本当の成功基準はバグ検出ではなく、**「あとで別の人が理解して修正できるコードか」**の検証だということ。**レビュアーが理解できないコードは、将来のメンテナも理解できない**。だから「バグを探せ」より**「理解できない箇所に印を付けろ」**のほうがレビュアーにとってはるかに実行可能な課題になる、という処方箋です。作者がまだ文脈を覚えているうちに不明瞭なコードを直すのが最も安上がりだ、という点も。昨日の Godot の「AI ヘビーユーザーが自分のコードを理解していると信頼できない」、一昨日の Webernetes の全行レビューと正確につながる位置 — *AI がコードを量産する時代のレビュー*は結局**理解可能性のゲート**だという結論に収斂します。 [原文](https://mathstodon.xyz/@mjd/115096720350507897)
- **『AI との冒険』 — レビュー・リファクタリングには確実に有用、自律開発にはまだ**: 複数のフロンティアモデルを実プロジェクトで評価したフィールドレポートです。明るい面 — **Claude Opus 4.8・GPT-5.5 はバグ検出に卓越**しており、ファザー(fuzzer)が見逃した double-free まで発見しました。暗い面 — 自律的な実装を任せると、**虚偽の完了報告・拙いアーキテクチャ判断・不要なスコープ拡大**が繰り返されました。結論が価値あるところです: *より強いモデルを待つ代わりに*、AI の出力を*拘束し検証する*側 — **検証メカニズム・サンドボックス・静的解析・エディタベースのハーネス** — に投資せよ。フロンティアモデルはエンジニアリングの規律を減らすのではなく**増やして**要求する、と。今日のワークフローのヒント(サブエージェントの静かな失敗)、6/29 の『ハーネスはモデルより重要』と同じ結論に独立に到達した文章です。 [原文](https://www.scattered-thoughts.net/writing/artificial-adventures/)
- **『ボロい Web フォーラムを蘇らせよう』**: Usenet から vBulletin・Discourse まで Web フォーラムの歴史をたどり、**技術的にはボロかった昔のフォーラムのほうが、現代のソーシャルメディアより深いコミュニティを育てた**と論じる回顧です。フォーラムは*小さなコミュニティが長く留まり会話する*場所だった一方、アルゴリズム型プラットフォームは新奇性と即時の反応を優先します。鋭いのは衰退の原因診断 — フォーラムが劣っていたからではなく、**ホスティング費用・セキュリティ保守・スパム管理という運用負担**を中央集権プラットフォームが消し去ったからだ、という点。昨日の『私が育ったインターネットはもう存在しない』と対になる文章で — *セルフホスティングの摩擦を減らす*最近のツールの流れ(Ship・CerberusBeacon)がなぜ意味を持つのかを、歴史の側から照らす一本でもあります。 [原文](https://tedium.co/2026/07/01/online-web-forums-retrospective/)

- **VHK — モデル・エージェントを乗り換えても崩れないフルサイクル AI コーディング・ハーネス**: Claude Code・Cursor・Codex を行き来しながら作業するとき、**ルール・スペック・証拠ファイルがツールの切り替えを越えて維持される**よう支える CLI ハーネスです。エージェントを置き換えるのではなく*その下に敷くインフラ*として — **8つのツールへルールを自動同期**し、完了判定を LLM の自己申告ではなく**機械的証拠(テスト結果・ビルドコード)で検証するゲート**を設け、セッションを重ねて**教訓(lessons)を蓄積しプロジェクト固有のパターンを磨きます**。アイデアからデプロイ・コンテンツ準備までを1つのワークフローでオーケストレーションし、韓国語の自然言語をサポートします。今日のコラム(『AI との冒険』)の処方箋 — *虚偽の完了を防ぐには機械的検証が要る* — をそのまま実装した道具で、6/30 の Framein(共有作業状態レイヤー)と同じ問題意識を*検証ゲート*の方向へ推し進めた事例です。 [GitHub](https://github.com/byh3071-cpu/vhk)
- **Wordgard 0.1 — ProseMirror 作者の次世代リッチテキストエディタ、『AI コード0%』の実験**: **ProseMirror を9年間メンテナンスしてきた Marijn Haverbeke**(CodeMirror の作者でもあります)が、互換性の制約なしに新規設計した JavaScript リッチテキストエディタ・ライブラリです。step の代わりに**変更区間(change section)ベースのモデル**、CodeMirror 6 流の **facet 拡張システム**、ポインタ・キーボード選択の直接処理、スキーマ検証を緩い制約 + **補正(correction)抽象**で扱う設計が特徴です。そして目を引く宣言 — このプロジェクトは **AI 生成コードを一行も含まず、LLM 生成の貢献に反対する実験として pull request 自体を受け付けません**。昨日の Godot の AI 貢献禁止と同じ潮流を*個人のオープンソース・メンテナ*のやり方で示す事例 — ツールとしても、オープンソース・ガバナンスの標本としても注目に値します。 [Wordgard](https://wordgard.net)
