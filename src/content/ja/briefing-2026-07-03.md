## 主要な新機能と実践活用

### Claude apps gateway — 『API キーのない Claude Code』を自社インフラで (6/29)

昨日 v2.1.198 の項で *Gateway への AWS アップストリーム追加* を短く扱いましたが、その『Gateway』の本体がこれです。**6/29、Anthropic が Claude apps gateway を発表**しました — **Amazon Bedrock・Google Cloud・Microsoft Foundry・Anthropic API の前段に立てる公式のセルフホスト型コントロールプレーン**で、別製品ではなく **`claude` バイナリに内蔵**されています(v2.1.195+)。ラップトップで Claude Code を動かすその実行ファイルが、サーバーではゲートウェイになります。

- **開発者の手元から API キーが消える**: 開発者はクラウドクレデンシャル・API キーの代わりに、**会社の IdP(OIDC — Okta・Entra ID・Google Workspace・Keycloak など)で SSO ログイン**し、短命の bearer トークンを受け取ります。**アップストリームのクレデンシャルはゲートウェイ(自社インフラ)にのみ**あり、IdP でアカウントを無効化すればセッション TTL(既定1時間)以内にアクセスが失効します — オフボーディングが IdP で完結します。
- **グループ別のモデル・ポリシー統制**: **IdP グループがモデル allowlist と managed settings ポリシーにマッピング**され、モデルアクセスは**サーバーサイドで強制**されます(許可外モデルへのリクエストは 400)。チームごとに異なるモデル・ツール・権限を配り、開発者はローカルで覆せません。
- **マルチクラウド・ルーティング + failover**: クライアントはゲートウェイに Anthropic Messages API で話し、ゲートウェイが **Bedrock・Google Cloud Agent Platform・Foundry・Anthropic API へ翻訳・ルーティング**します。リージョン・プロバイダー・failover 順を変えても、開発者側の再設定は不要です。
- **データは自社側に残る**: **Anthropic API をアップストリームにしない限り、ゲートウェイのデータプレーンは Anthropic のインフラへ何も送りません。** テレメトリは OTLP で自社のコレクター(Datadog・Splunk・ClickHouse など)へ流れ、ゲートウェイはプロンプト・応答の内容を保存しません。

```yaml
# gateway.yaml — 最小構成(この5セクションでサインインのループが回る)
listen:
  host: 0.0.0.0
  port: 8080
  public_url: https://claude-gateway.internal.example.com   # プライベートネットワークのアドレスのみ許可
oidc:
  issuer: https://login.example.com
  client_id: 0oa1example2
  client_secret: ${OIDC_CLIENT_SECRET}
  allowed_email_domains: [example.com]
session:
  jwt_secret: ${GATEWAY_JWT_SECRET}
  ttl_hours: 1            # IdP で無効化されたユーザーはこの時間内に失効
store:
  postgres_url: ${GATEWAY_POSTGRES_URL}
upstreams:
  - provider: bedrock
    region: us-east-1
    auth: {}              # AWS 既定のクレデンシャルチェーン(IRSA・タスクロール等)
```

```bash
claude gateway --config gateway.yaml
# 開発者側は MDM で forceLoginMethod: "gateway" + forceLoginGatewayUrl を配布すれば
# /login が直接 Cloud gateway 画面で開く
```

要点は、**『データレジデンシーやガバナンスの理由で、推論を自社クラウド経由にする必要がある組織』のための公式ルート**だということです。6/30 の Foundry GA(Azure のガバナンスそのまま)、7/2 の anthropicAws アップストリーム・failover と正確に同じ流れ — *Claude Code のエンタープライズ展開が『個人ツールの組織内拡散』から『IdP・ポリシー・コスト統制の付いた管理型プラットフォーム』へ*移る構図です。制約も明確なので導入前に確認を — ゲートウェイセッションでは **WebSearch が無効**、プロンプトキャッシュは **5分 TTL のみ**、**CI 用のサービストークンフローなし**(サインインは常に人間のブラウザ・デバイスフロー)、サーバーは **Linux 専用・OIDC 専用**です。[発表ブログ](https://claude.com/blog/introducing-the-claude-apps-gateway) · [公式ドキュメント](https://code.claude.com/docs/en/claude-apps-gateway)

### Enterprise のモデル entitlements(beta)— モデルだけでなく『effort』まで組織が統制 (7/1)

**7/1、Enterprise プランにモデル entitlements が beta で**入りました — 管理者が**ユーザーがどのモデルにアクセスできるか**、そして**どの effort レベル設定を使えるか**を組織単位で統制します。

注目は **effort まで統制対象**という点です。7/1 のブリーフィングで扱ったとおり、Sonnet 5 は **effort が high 既定**で、*トークン単価は安くてもタスクあたりの消費が膨らみうる*構造です — これまでそのチューニング(`--effort medium` など)は個人の習慣任せでした。これからは**『組織の既定は medium、high は特定グループのみ』**のようなルールを管理者がポリシーとして強制できます。6/29 の組織既定モデル(`Org default`)、上の apps gateway のグループ別 allowlist と同じ場所 — *モデル・コストのガバナンスが個人設定から組織のポリシー層へ*上がる流れの延長です。Enterprise チームなら、Sonnet 5 移行とあわせて effort ポリシーをこの機能で固定しておきましょう。[Claude リリースノート](https://support.claude.com/en/articles/12138966-release-notes)

## 開発者ワークフローティップス

### Webernetes の教訓 — 『AI が8割書いたコード』はレビューだけでも、テストだけでも足りない

ブラウザで動く Kubernetes 実装 **Webernetes**(ngrok のエンジニアリングチーム)が、良質な実測データを公開しました。**Claude が2ヶ月間で552コミット・629ファイルにわたりコードベースの約80%を生成**しましたが、成果物が堅牢だった理由はモデルではなく**検証体制**でした — ① **人間が全行をレビュー**し、② **統合テスト204本 + ユニットテスト1,855本**を維持し、③ 挙動を**実際の k3s クラスタと対照検証**したのです。

記事が挙げる **LLM の反復ミス3種**は、そのまま監視リストに使えます:

```text
# AI 生成コードで繰り返し捕捉されたミスのパターン(Webernetes の実測)
1. 省略実装 — 仕様の一部だけ実装し、完成品のように見せて終える
2. 不要なヘルパー関数 — 既存コードを探さず新造して重複を生む
3. 欠落テストケース — ハッピーパスだけ覆い、境界条件を落とす
```

結論は **『コードレビューだけでも、テストだけでもなく、両方』**です — レビューは構造・重複(2)を捕らえ、テストは省略実装・欠落(1・3)を捕らえます。片方だけでは残り半分が素通りします。6/28 の**作業者と採点者の分離**、6/29 の**ハーネスがモデルより重要**と同じ場所で — *人間の審美眼・理解と LLM の速度の組み合わせ*がどこまで届くか(約140KiB に圧縮されたブラウザ K8s)と、そのとき人間が握り続けるべきものを同時に示す事例です。[ngrok ブログ](https://ngrok.com/blog/i-ported-kubernetes-to-the-browser) · [GeekNews](https://news.hada.io/topic?id=31014)

### 復帰した Fable 5 — 『どのモデルが実際に応答したか』をログに残す

Fable 5 は戻ってきましたが、下のセキュリティ節で扱うとおり、**ガードレールに触れたリクエストは Opus 4.8 へルーティング**されます(通知は表示されます)。これに v2.1.198 の **model-not-found failover**(次のプロバイダーへ自動進行)、6/29 の**組織既定モデル**まで重なると — *指定したモデルと実際に応答したモデルがずれうる経路*が、ここ数日で3つに増えました。

```text
# 『指定したモデル ≠ 応答したモデル』になりうる経路(2026-07 時点)
1. Fable 5 のガードレールブロック → 応答は Opus 4.8 へ (7/1〜)
2. Gateway の failover → model-not-found で次のプロバイダーへ (v2.1.198)
3. 組織ポリシー → Org default・entitlements が選択を上書き (v2.1.196〜)
```

実務の処方箋は2つです。① **パイプラインのログに実際にサービングしたモデルを記録**しましょう — 出力品質が揺れたとき、*モデルが変わったターン*を特定できなければデバッグになりません(OpenTelemetry メトリクスにはモデルが含まれます)。② 再現性が重要なクリティカルな自動化は、**明示的なモデル固定 + `fallbackModel`** で意図をコードに残しましょう — 7/7 までは Fable 5 が週次上限の50%キャップの下にあるため、上限消化時の代替経路まで含めてです。7/1 の**2段構成(既定は Sonnet 5、頂点は Opus 4.8)**と同じ場所 — *モデル選択がランタイムに動く時代*には、可観測性がそのまま再現性です。[Fable 5 再展開の告知](https://www.anthropic.com/news/redeploying-fable-5) · [@claudeai](https://twitter.com/claudeai/status/2072402636813607381)

## セキュリティ・制限事項

### Fable 5 復帰の続報 — ガードレールは『より強く』戻ってきた: 過剰ブロックの可能性と Opus 4.8 fallback (7/1〜)

昨日扱った復帰の**実運用の続報**です。Anthropic は米政府との協議に沿って **Fable 5 のサイバーセキュリティ・セーフガードを更新**して再展開しました — そしてそのガードレールは**以前より敏感に**設定されています。

- **過剰ブロックの可能性**: ほとんどのコーディング作業には影響しませんが、**当初は無害(benign)なリクエストも以前より頻繁に引っかかる**可能性があります。リクエストがブロックされると**明確な通知が表示され、応答は Opus 4.8 にルーティング**されます — セッションが死ぬのではなく、モデルが替わります(上のワークフローティップス参照)。
- **bio/chem 分類器は据え置き**: 生物・化学の分類器はリリース初期の状態のままで、**基礎科学の質問でも fallback 応答が発生しえます**。Anthropic は**数週間かけた改善**を予告しました。
- **コミュニティの反応**: Hacker News では、**正当なクエリまで塞ぐ過剰フィルタがモデルの効用を削る**、**予測不能なガードレールに長い作業セッションを預けられない**、**価格(7/7 以降のクレジット移行)の不確実性**といった不満が出ており、競合モデルへ移ったという報告もあります。

要点は **『復帰したが、以前のままではない』** — 6/30 のブリーフィングの警告(*戻っても現状復帰と仮定するな*)がそのまま現実になりました。セキュリティ研究・低レイヤーのシステム作業などガードレールに近いドメインなら、Fable 5 をパイプラインに戻す前に**代表的なプロンプト数本でブロック率を実測**しましょう。輸出規制解除の見返りに約束された*政府協力の枠組み*(7/2 ブリーフィング)が、こうして**モデルの日常動作にまで降りてきた**最初の事例でもあります。[GeekNews](https://news.hada.io/topic?id=31035) · [@claudeai](https://twitter.com/claudeai/status/2072402636813607381)

### 7/2 未明の Opus 4.8 elevated errors — 今月のインシデント・パターンが継続 (7/2)

**7/2 の 00:38〜01:19 UTC(約41分)**、**Opus 4.8 の elevated errors** インシデントがありました — 影響範囲は claude.ai・Claude API・**Claude Code**・Claude Cowork です。6/30 にも同モデルの elevated error rate(14:31〜15:28 UTC)があり、6月を通して続いた**短く反復する安定性のノイズ**は7月にも続いています。上のワークフローティップスの fallback ルーティングと重ねて見ると — Fable 5 のブロック時に応答するモデルはまさに Opus 4.8 なので、**fallback 経路そのものも揺れうる**ことまで織り込み、無人実行ではリトライ・バックオフ・失敗通知の基本を維持しましょう。[Claude Status — Incident History](https://status.claude.com/history)

### ZCode 3.0 — モデルベンダーがハーネスまで: Zhipu AI の GLM-5.2 公式フレームワーク

6/29 のブリーフィングで **Semgrep ベンチマークの GLM 5.2**(オープンウェイトがサイバー検知で Claude を僅差で上回った)を扱いましたが、その Zhipu AI が今度は **GLM-5.2 の公式ハーネス ZCode** を自ら出荷しました。デスクトップアプリ(macOS・Windows・Linux beta)で **planning → coding → review → deployment** を一つのワークフローに統合し、長期タスクは **Goals** で管理、**WeChat・Feishu・Telegram ボットでのリモート制御**まで備えます。サブスクリプションは**月16.2〜144ドル**の3段階です。

要点は**『モデルベンダーがハーネスまで垂直統合する』流れ**です — Anthropic の Claude Code、OpenAI の Codex に続き、オープンウェイト陣営の Zhipu も*モデル+ハーネスのセット*で勝負します。6/29 の Semgrep の結論(*性能差はモデルよりハーネスで生まれる*)を、モデルベンダー自身が実行に移した格好です。コミュニティはインターフェースが Claude Code より **Codex に近い**と評しました — 6/28 のアジアの代替モデルの流れとあわせて、*エージェンティック・コーディングの競争がモデル単品からハーネス生態系へ*移るシグナルとして読めます。[ZCode](https://zcode.z.ai/en) · [GeekNews](https://news.hada.io/topic?id=31047)

## コミュニティニュース

- **Godot エンジン、AI 生成コードの貢献を禁止 — 『AI は責任を取れない』 (7/1)**: オープンソースのゲームエンジン **Godot** が貢献ポリシーを改定し、**AI が書いたコードの貢献を事実上禁止**しました。新ポリシーの骨子は — **コードは人間が書く**こと、AI の補助は**些末なもの(補完・正規表現・置換)に限定**、AI を使った場合は **PR での開示(disclosure)が義務**。AI エージェントが提出する PR や、人間同士のコミュニケーションでの AI 生成テキストも禁止です。根拠が鋭い — **『PR を作るコストは急落したのに、レビューするコストとレビューできる人は据え置き』**という非対称、そして *『AI は責任を取れないし、AI のヘビーユーザーがレビューのフィードバックに意味のある応答ができるほど自分のコードを理解していると信頼できない』*ということ。上の Webernetes(80% AI コードを全行レビュー・テストで支えた事例)と正確に対をなします — **AI コードの品質の問題ではなく『誰が責任を持って保守するのか』の問題**であり、オープンソース生態系がエージェンティック・コーディング時代のルールを書き直し始めたシグナルです。AI 補助でオープンソースに貢献するなら、プロジェクトごとのポリシー確認と開示がこれからの基本動作です。[Godot 公式発表](https://godotengine.org/article/contribution-policy-2026/) · [The Register](https://www.theregister.com/ai-and-ml/2026/07/01/godot-says-bye-bye-ai-bans-vibe-coded-contributions/5265344) · [PC Gamer](https://www.pcgamer.com/gaming-industry/open-source-game-engine-godot-will-no-longer-accept-ai-authored-code-contributions-we-cant-trust-heavy-users-of-ai-to-understand-their-code-enough-to-fix-it/)

## 知っておくと便利な小さな変更点

今日扱った事案の実用的なディテールです。

- **Fable 5 の上限リマインド**: 有料サブスクライバーは **7/7 まで週次上限の最大50%**で Fable 5 を使え、以降は**別途の使用量クレジット**が必要です
- **apps gateway の要件**: Claude Code **v2.1.195+**(サーバー・クライアント両方)、**PostgreSQL 14+**、サーバーは **Linux 専用**(macOS はローカル開発のみ、Windows 非対応)、認証は **OIDC 専用**(SAML・LDAP 非対応)
- **gateway セッションの機能制約**: **WebSearch 無効**(アップストリームの対応を確認できないため)、プロンプトキャッシュは **5分 TTL のみ**(1時間 TTL beta は対象外)、初回接続時に **TLS 証明書のフィンガープリントを固定**(証明書ローテーションは計画イベントとして扱う)
- **gateway の支出上限**: 支出上限を**組織・グループ・ユーザー単位**で設定でき、暴走したワークロードがコミットメント全体を焼き尽くすのを防ぎます
- **モデル entitlements は beta**: Enterprise 専用で、モデルアクセスとともに **effort レベル設定まで**統制対象です
- **bio/chem 分類器の改善予告**: 基礎科学の質問での fallback 誤検知は**数週間かけて改善**される予定です

## 注目プロジェクト＆ツール

- **naeryeo — 韓国の公共交通の経路・費用を AI に: CLI + MCP サーバー**: 韓国の**公共交通の経路と費用**を照会する CLI であり **MCP サーバー**です。`naeryeo route --from --to` で経路を出し、`naeryeo mcp` で MCP サーバーを立てれば、Claude のようなエージェントが*通勤距離・交通費の計算*を直接こなします — 作者が実際に**住まい探しで通勤コスト計算を自動化**するために作ったという出自に説得力があります。設計のディテールも良い: 経路は ODsay、ジオコーディングは Kakao と**国内 API 2つをハイブリッド結合**し(どちらか単体では完結しない隙間を埋める)、**API キーは OS のキーチェーンに保存**(`naeryeo setup`/`logout`)。Homebrew・Scoop・npm でインストールできます。*ドメイン特化の MCP サーバーが実生活の意思決定に届く*きれいな事例です。[GitHub](https://github.com/GyeongHoKim/naeryeo)
- **Ship — `ship --service demo` 一発でローカルサービスをカスタムドメインへ**: ホームサーバーや Mac mini で動くローカルウェブサービスを**カスタムドメインへ素早くデプロイ**するオープンソースツールです。SSR プロジェクトに Dockerfile を足して `ship --service demo` を打てば、`demo.your-domain.com` で開きます。既定は **Tailscale 上のプライベートアクセス**で、必要なサービスだけ選んで公開インターネットに露出します — デプロイ済みサービス・内部/外部の状態・ログ・実行コマンドを見るダッシュボードもあります。まだ初期バージョンで Cloudflare ドメインと Tailscale の設定が必要ですが、*小さなサービスを複数運用する*セルフホスターの摩擦(ポートフォワーディング・手作業のリバースプロキシ)を正面から減らすアプローチです。6/28 の CerberusBeacon(必要なときだけ開くトンネル)と同じ流れの — *常時露出なしのセルフホスティング*です。[GitHub](https://github.com/gronxb/ship)
