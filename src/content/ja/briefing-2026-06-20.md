## 主要な新機能と実践活用

### auto モード安全ガード — 依頼していない「破壊的コマンド」をランタイムが止める（v2.1.183）

今回のリリースの目玉です。**auto モード**（操作ごとの承認を求めずツール実行を自動許可するモード）で、エージェントが**取り返しのつかない破壊的コマンド**を、ユーザーが明示的に依頼していない場合に**ランタイムがブロック**します。対象は具体的です。

- **破壊的 git コマンド**: `git reset --hard`、`git checkout -- .`、`git clean -fd`、`git stash drop` — **「ローカル作業を捨てて」と依頼していなければ**ブロックします。
- **`git commit --amend`**: **このセッションのエージェントが作っていないコミット**を書き換えようとするとブロックします（他人の履歴の上書き防止）。
- **インフラ破棄**: `terraform destroy` / `pulumi destroy` / `cdk destroy` — **特定のスタックを指定して依頼していなければ**ブロックします。

```bash
# auto モードでエージェントが勝手にこれを実行 → これからはブロックされる
git reset --hard origin/main
terraform destroy

# 本当に必要なら「明示的に」意図を伝えれば通る
# 例:「staging スタックだけ destroy して」「ローカル変更を捨てて origin/main に合わせて」
```

実務上の意味は明確です。6/15 の**クレジット分離**以降、auto モードで長い作業を無人実行するパターンが増えましたが、その最大のリスクが**エージェントが推測で破壊的コマンドを実行し、成果物やインフラを消してしまうこと**でした。この種の操作が、**「ユーザーが明示的に依頼したか」**を基準にランタイムで一度フィルタされるようになります — 6/17 で扱った「取り返しのつかない箇所だけ人間のゲートを置く」設計を、今回は**ツールがデフォルトで**提供する形です。加えて、**要求したモデルが deprecated、あるいは自動的に新しいモデルへ置き換えられた**場合、`-p`（print）モードの **stderr** に警告が出るようになり、**agent frontmatter に固定した model も**その警告対象に含まれます。[GitHub v2.1.183](https://github.com/anthropics/claude-code/releases/tag/v2.1.183)

## 開発者ワークフローティップス

### auto モードの無人実行は安全になった — それでも「ブランチ/worktree」で影響範囲を絞る

上記ガードのおかげで auto モードを無人で回す負担は下がりました。ただし注意書きを読んでください — ガードが止めるのは**「明示的に依頼していない破壊的コマンド」**であって、**あらゆる事故**ではありません。エージェントが*もっともらしい理由をつけて*大量にファイルを書き換えたり、ブロックリストにない方法でミスをする余地は残ります。ですからレシピは従来どおりです: **専用の feature branch または worktree の中で** auto モードを回し、結果が良ければ人間が PR でマージします。

これは 6/17 で扱った **「AI は feature branch にのみ push、マージは人間が PR で」** のゲートとちょうど対になります — ランタイムガード（破壊的コマンドのブロック）が**一枚目の安全網**、ブランチ/worktree の隔離が**二枚目**です。二層を揃えれば、auto モードを高速に回しつつ **取り返しのつかないミスをメインラインに届かせない**ことができます。今回のガードはその一層目をデフォルトで敷いただけで、隔離まで肩代わりはしてくれない — そこは引き続き自分の責任だと前提にワークフローを設計してください。[GitHub v2.1.183](https://github.com/anthropics/claude-code/releases/tag/v2.1.183)

### モデル自動アップグレード警告を「CI の stderr」で拾い、agent frontmatter に model を固定する

v2.1.183 のモデル警告は単なる案内ではなく、**ヘッドレス自動化の落とし穴を一つ塞ぐレバー**です。CI・cron で `claude -p` を回すパイプラインは、指定したモデルが**静かに deprecated されたり新モデルへ自動置換されたり**すると、ある日突然 **出力品質・トークン消費・挙動が変わる**ことがあります。その置換が **stderr 警告**として表面化するようになったので、パイプラインで **stderr を捨てずにログに収集**しておけば、「モデルが変わった」というシグナルを見逃しません。

```yaml
# .claude/agents/reviewer.md — サブエージェントの model を明示的に固定
---
name: reviewer
description: 変更ファイルのバグ・規約違反のみをレビュー
tools: Read, Grep
model: claude-sonnet-4-6   # この行が deprecated になれば v2.1.183 が警告してくれる
---
```

要点は二拍子です — **①agent frontmatter に `model:` を明示**してどのモデルで動くかをコードに固定し（6/14 の「サブエージェントの model を repo にコミット」と同じ場所）、**②CI で stderr 警告をモニタ**して自動置換を早期に検知すること。輸出規制でモデル可用性が揺れた最近の流れと重ねて、`fallbackModel`・`availableModels` で代替経路を敷く習慣とあわせ、**「自分の自動化が正確にどのモデルで動くか」**を可視化しておけばドリフトを防げます。[GitHub v2.1.183](https://github.com/anthropics/claude-code/releases/tag/v2.1.183)

## セキュリティ・制限事項

### Claude 全プラットフォーム障害 — Chat・Web・Claude Code が同時ダウン（6/18、06:55〜07:40 UTC）

6/18 の朝、**Claude 全プラットフォームが約 45 分間の障害**に見舞われました。**06:55〜07:40 UTC** の間に **Claude Chat・Web インターフェース・Claude Code** が同時に影響を受け、ユーザーは **サーバータイムアウト・応答なし・ログインエラー**を経験しました — チャットが読み込まれない、プロンプトが実行されない、応答が遅延する、といった症状です。Downdetector では報告が急増し、Anthropic は **claude.ai に影響した原因を特定して復旧**したと発表しました。（本稿執筆時点で公式のポストモーテムは未公開です。）

## エコシステム＆プラグイン

### Anthropic の管理型 MCP connector — 管理者が一度承認すれば、社員は初回ログインで zero-touch 継承（6/18）

Anthropic が **Enterprise-Managed Authorization（EMA）** を出し、**MCP connector の認証を管理者が IdP で中央管理**できるようにしました。これまで MCP のエンタープライズ普及を阻む最大の壁が、**connector ごとにユーザーが個別に OAuth 同意を押す必要がある**ことでした — セキュリティチームが中央で統制できないこの手順を、EMA がなくします。

- **zero-touch プロビジョニング**: 管理者が connector を**一度承認**すれば、ユーザーは既に持つ **IdP のグループ・ロールで接続を継承**し、**初めて Claude を開いたときには connector が既にそこにあります**。新入社員は初日に IT へチケットを切らずとも、**承認済みの connector をすべて**手にできます。
- **Claude Code を含む**: 管理者は MCP サーバーを **Claude・Claude Code・Cowork** 全体で承認でき、同じ zero-touch が**ターミナルの Claude Code にも**そのまま適用されます。
- **Okta が最初の IdP**: Okta の Cross App Access（XAA）でプロビジョニングし、**Asana・Atlassian・Canva・Figma・Granola・Linear・Supabase** が既に EMA に対応（Slack ほか追加中）。**Team・Enterprise** プラン向けのベータです。

実務上は **「connector を一度設定すればチーム全員が認証なしで使える」** ということです — 個々の開発者が MCP サーバーごとに OAuth を回していた摩擦が、組織単位では管理者設定一回で消えます。下のコミュニティ項目にある **MCP 標準レベルの変化**が、この製品機能の土台です。[Claude ブログ — Enterprise-managed auth](https://claude.com/blog/enterprise-managed-auth) · [Okta ニュースルーム](https://www.okta.com/en-ca/newsroom/press-releases/okta-becomes-a-featured-identity-provider-powering-secure-ai-agent-connections-for-claude-enterprise/)

## コミュニティニュース

- **MCP 標準の Enterprise-Managed Authorization が「stable」到達 — Anthropic を超えて広がる zero-touch 認証（6/19）**: 上記 Anthropic の製品機能の**土台である MCP 標準そのもの**が、今週重要な分岐点を越えました。**Model Context Protocol の Enterprise-Managed Authorization 仕様が stable に昇格**したことで、zero-touch OAuth は **Anthropic 一社の機能ではなく標準の一部**となり、報道によれば **Okta のような IdP、VS Code のようなクライアント**までが同じ仕組みを採用し始めています。MCP コミュニティが**「エンタープライズ規模の MCP 採用における単一最大の障害」**と呼んできた*ユーザーごとの認証手順*が、**個別ベンダーの回避策ではなくプロトコル標準として**解消される流れです。開発者への含意は明確で、MCP が「個人がローカルで OAuth を回す実験段階」を超え、**セキュリティチームが中央でガバナンスするエンタープライズ基盤**へと固まりつつあるというシグナルです。（Claude Code での*使い方*は上のエコシステム項目をご覧ください。）[MCP ブログ — Enterprise-managed auth](https://blog.modelcontextprotocol.io/posts/enterprise-managed-auth/)

- **起動ロゴ下の「setup issues」行を削除**: 設定の問題は今後 **`/doctor`** または `--debug` で確認します（v2.1.183）

## おすすめコラム＆読み物

- **「ローカル Qwen は劣る Opus ではなく、別のツールだ」（Alex Ellis）**: ローカル Qwen が「ほぼ Sonnet/Opus 級」という主張に実測で答える記事です。**Qwen 3.6 27B の SWE-bench Verified は 77.2、Claude Opus 4.8 は 88.6%** — 真っ向勝負では劣りますが、著者の主張は **「等級が低いのではなくカテゴリが違う」** ことです。ローカルモデルは **プライバシー重視の作業・固定費・ベンダーリスク回避**で価値を発揮する一方、反復出力ループとハルシネーションのため **長時間の無人実行には不向き** — 実際のソフトウェア事業での運用経験に基づく結論です。すなわち「より安い Opus の代替」ではなく、**カスタマーサポートや狭い保守といった特定タスクの別のツール**として使え、と。クレジット分離後に「何をローカルで、何をクラウドで」を線引きする実用的な視点です。[Alex Ellis' Blog](https://blog.alexellis.io/local-ai-is-not-opus/)
- **「.gitignore だけが Git でファイルを無視する方法ではない」（Nelson Figueroa）**: git の無視ルールが**共有範囲に応じて 3 つのスコープ**に分かれる、小さいがよく見落とされる実務知識です — ①**`.gitignore`**（リポジトリにコミットされ、チーム全体で共有するルール）、②**`.git/info/exclude`**（コミットされない **repo ローカルの個人ルール** — 例: 自分だけの `notes.txt`）、③**`~/.config/git/ignore`**（`core.excludesFile` で設定する **マシン全体のルール** — OS・IDE の生成物など）。要点は **OS・エディタ固有の項目をプロジェクトの `.gitignore` に詰め込まず**、ユーザーレベル設定へ逃がすこと — プロジェクトファイルをきれいに保ちつつ、自分のすべてのローカルリポジトリに一貫して適用されます。AI エージェントが生成する一時ファイルが増える昨今、共有用と個人用の無視ルールを分けておく習慣はますます価値があります。[Nelson Figueroa](https://nelson.cloud/.gitignore-isnt-the-only-way-to-ignore-files-in-git/)

## 注目プロジェクト＆ツール

- **mdpresent — Markdown を LLM なしで PPTX/HTML/PDF スライドにレンダリング**: Markdown ドキュメントを **プレゼンスライド（PPTX・HTML・PDF）** に変換する CLI ツールですが、**外部 LLM を使いません**。Markdown を **「Presentation IR」→「Layout IR」** の 2 段階の中間表現に変換してからレンダリングし、**見出し・密度・リスト数**といった構造要素を基準にスライドを賢く分割します。出力 PPTX は **リスト・強調・ダイアグラムといった意味的書式を保ったまま編集可能**で、複数のデザインテーマに対応し、**Markdown を人間が編集し続ける原本（authoritative source）として**維持します。AI が下書きを出しつつ、成果物は決定的（deterministic）かつ再現可能であるべきワークフローにぴったりです。[GitHub](https://github.com/ch040602/MdPr)
- **htmlbook — AI が作った HTML をファイルでなく「リンク 1 つ」で共有**: AI が生成した **HTML・markdown ファイルを公開リンクに**変換し、添付ファイルを送ったりコードを貼り付けたりせずに共有できるサービスです。`.md`・`.html` をアップロードするか **MCP で接続**すると、**モバイル・デスクトップで開ける URL** が即座に生成されます（公開、またはワークスペースメンバー限定）。要点は **markdown 原本を保存**すること — エージェントが同じリンクを壊さずに**内容を再訪して更新**できます。Claude・エージェントが作ったレポートやダッシュボードを「ファイルでやり取りする」摩擦なしに渡すのに実用的です。[htmlbook.io](https://htmlbook.io)
