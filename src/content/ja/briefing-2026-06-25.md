## 主要な新機能と実践活用

### Claude Tag — Slack チャンネルに「共有される AI チームメイト」として入る (6/23, ベータ)

Anthropic が **Claude Tag** を公開しました（6/23, ベータ）。要点は、Claude を Slack チャンネル内の **持続的で共有される AI チームメイト** にすることです — 誰でも `@Claude` でタスクを渡すと、Claude が **数時間〜数日かけて段階的に非同期で**処理し、結果をスレッドに投稿します。チャンネルの全員が **何を作業中か見え**、別の人が止めた地点から会話を **引き継げます**。

- **チャンネル単位の共有メモリ**：Claude Tag はチャンネル全体に **1つの共有 Claude** を与え、チャンネルの活動から **メモリを蓄積**します — 許可すれば **他の Slack チャンネル・データソースからも自動的に学習**します。
- **ambient モード**：有効にすると Claude が **必要そうな情報を先回りで**通知します — 尋ねる前にアップデートを出す形です。
- **アイデンティティ・アクセスの分離**：管理者は **用途ごとに分離した Claude のアイデンティティ** を作れ、「セールス Claude」と「エンジニアリング Claude」がメモリを共有しません。**接続するツール・データ・コードベース** と **支出上限** まで、管理者が細かく制御します。
- **モデル・対象**：**Opus 4.8** 上で動作し、現在は **Claude Enterprise・Team** プラン向けベータです。

```text
# Slack チャンネルでそのまま委任する
@Claude このインシデントスレッドを整理して、タイムラインとフォローアップ項目にして
@Claude この PR をレビューして、詰まったら ambient で知らせて
```

実務上の意味は明確です — Claude が **ターミナル・IDE・Web** を越え、今度はチームが実際に集まる **Slack チャンネル** の中に入ったということです。6/22 で扱った **Artifacts**（セッション作業を社内共有ページに）が *成果物の共有* だったとすれば、Claude Tag はその上で **エージェント自体を協働空間に常駐させる** 一歩です — 「一人がローカルで回す道具」から「チャンネルが共有する同僚」へと重心が移ります。なお **既存の `Claude in Slack` アプリは 8/3 で終了** し、Claude Tag が置き換えるため、そのアプリを使っていたチームは移行対応が必要です（下記エコシステム）。[Anthropic — Introducing Claude Tag](https://www.anthropic.com/news/introducing-claude-tag) · [TechCrunch](https://techcrunch.com/2026/06/23/anthropics-claude-tag-is-learning-your-company-one-slack-message-at-a-time/)

### `/rewind` が `/clear` の前まで遡って復元できるようになった (v2.1.191)

小さいながら **ミスを取り消せる** 変更です。`/rewind`（会話・コード状態を以前の時点に巻き戻す機能）が、**`/clear` を実行する *前* の会話まで** 遡って復元できるようになりました。

```text
# 誤って /clear して文脈をまるごと消したとき
/rewind   # /clear 前の時点まで戻って会話を救える
```

要点は **`/clear` がもはや「取り返しのつかない崖」ではない** ということです — 長いセッションでコンテキストを整理しようと `/clear` を押し、肝心の文脈まで消してしまったとき、`/rewind` でその前まで巻き戻せます。無人自動化よりも、**対話セッションを長く引っ張る** 人に直接効くセーフティネットです。[GitHub v2.1.191](https://github.com/anthropics/claude-code/releases/tag/v2.1.191)

## 開発者ワークフローティップス

### カンマでまとめた hook matcher が「静かに発火しない」バグが直った — ガード hook を再検証せよ

CLAUDE.md のルールを **hook で強制** している人は、この修正を必ず押さえてください。v2.1.191 は **カンマ区切りの hook matcher**（例：`"Bash,PowerShell"`）が、これまで **静かに一度も発火しない** バグを修正しました。つまり、複数のツールを1つの matcher にカンマでまとめてガードを掛けていた場合、**その hook は実は死んでいた可能性があります。**

```jsonc
// .claude/settings.json — カンマでまとめた matcher は v2.1.191 まで発火しなかった可能性
{
  "hooks": {
    "PreToolUse": [
      { "matcher": "Bash,PowerShell", "hooks": [ /* ... */ ] }
    ]
  }
}
```

これは 6/22 で扱った **「必ず守るべきルールはプロンプトではなく hook で強制せよ」** とちょうど対になります — *その hook が実際に発火している保証* がなければ、強制も存在しません。カンマでまとめた matcher を使っているなら、**①更新し、②危険なコマンドで hook が実際にブロックするか一度再現** してください。（不安ならツールごとに matcher を分けるのが最も確実です。）[GitHub v2.1.191](https://github.com/anthropics/claude-code/releases/tag/v2.1.191)

### 無人実行の MCP 信頼性が一段上がった — リトライ + ヘッドレス OAuth

CI・cron で Claude Code を無人実行し MCP サーバーを接続しているなら、v2.1.191 の MCP 改善はそのまま効きます。**①capability discovery が一時的なネットワークエラーで短いバックオフ付きでリトライ**し、**②MCP OAuth の discovery・トークン要求も一時エラー後に1回リトライ**し、**③ヘッドレス環境ではブラウザのポップアップを飛ばして直接 URL 貼り付けプロンプトに**進みます。さらに **MCP の 404 エラーは URL と MCP 設定の場所を表示** するため、原因特定が速くなります。

要点は **「1度のネットワークの揺れが無人作業全体を壊す」摩擦を減らす** ガードだということです — 6/24 で扱った **`CLAUDE_CODE_MCP_TOOL_IDLE_TIMEOUT`**（止まったリモート MCP 呼び出しを無限ハングではなく自動中断）と同じ方向の一歩です。両方を併せれば、CI でリモート MCP が一時的に揺れても **リトライで補うか、きれいに失敗** します。無人で MCP を使うパイプラインなら更新を。[GitHub v2.1.191](https://github.com/anthropics/claude-code/releases/tag/v2.1.191)

## セキュリティ・制限事項

### 6/24 にもう一度インシデント — Opus 4.8 の elevated error rate（今週4度目）

6/24、**Claude にもう一度インシデント** がありました。集計サイトの記録では、**6/24 午後（約 13:18 UTC）に Opus 4.8 の elevated error rate** が始まり、**約1時間40分** 続いた後に復旧しました（公式 status ページが確認時点で読み込めず、時刻・長さは集計サイト基準です）。

個別のインシデントは復旧しましたが、**パターンが信号** です — 6/18 の全面障害、6/20 の 'Opus 4.8 errors'、6/23 の複数モデルのインシデントに続き、**今週だけで4度目** の安定性のぐらつきです。ターミナルの Claude Code も同じバックエンドに紐づくため、短い障害でも無人作業がまるごと壊れ得ます。**リトライ・バックオフ・失敗通知** をパイプラインに仕込むのが変わらぬ実務対応です（6/22 の `CLAUDE_CODE_MAX_RETRIES` 上限15・`RETRY_WATCHDOG` と併せて点検を）。加えて **Fable 5・Mythos 5 は輸出規制13日目でも依然として全ユーザーにオフライン** です — 外国籍の開発者（韓国・日本など）の対応は変わりません：`fallbackModel` で **Opus 4.8 のような代替経路** を用意しておきましょう。[Claude Status — Incident History](https://status.claude.com/history) · [StatusGator](https://statusgator.com/services/claude/claude-code)

## エコシステム＆プラグイン

### 既存の `Claude in Slack` アプリ、8/3 で終了 — 30日の移行期間（管理者対応が必要）

上の Claude Tag 公開と対になる **サポート終了の予告** です。既存の **`Claude in Slack` アプリが 8/3 で引退** し、Claude Tag が **全面的に置き換え** ます。自動では移行しないため、**管理者の対応が必要** です。

- **30日の期間**：管理者は移行期間（約30日）内に **Claude Tag へ opt-in** し、**チャンネルアクセス・ツール接続・メモリ範囲** を 8/3 の締め切り前に設定する必要があります。
- **なぜ重要か**：Claude Tag はアイデンティティ・メモリ・ツールアクセスのモデルが旧アプリと異なり、設定が *引き継がれません* — 移行しなければ 8/3 以降、旧アプリの Slack 統合が止まります。

実務上の読み方は単純です — チームが `Claude in Slack` を使っていたなら、これは「新機能の見物」ではなく **締め切りのある移行作業** です。6/20 で扱った **管理型 MCP connector**、6/23 で扱った **組織単位のモデル制限** と同様、Anthropic が *管理コンソールで統制するエンタープライズ面* を広げ続ける一場面です — 導入も終了も、いまや管理者設定に紐づきます。[Anthropic — Introducing Claude Tag](https://www.anthropic.com/news/introducing-claude-tag)

## 知っておくと便利な小さな変更点

v2.1.191 で上記に入りきらなかった実用的な変更・修正です。

- **ストリーミング中の CPU 使用 約37%減**：テキスト更新を **100ms 単位でコアレッシング** し、ストリーミング応答中の CPU 使用を約37%削減、**長時間セッションのメモリ増加**（ターミナル出力キャッシュ）も緩和されました — ノート PC のファンが回っていた長時間セッションに体感できる変化です (v2.1.191)
- **background agent の停止が「恒久的」に**：タスクパネルからエージェントを停止すると、もう **復活しなくなりました**（以前は停止後に蘇る問題がありました）(v2.1.191)
- **`/voice` が制限理由を表示**：組織ポリシーで `/voice` が無効なとき、一般的な 'not available' ではなく **何によって制限されているか** を説明します (v2.1.191)
- **`/permissions` の Recently-denied の承認が保持**：'Recently-denied' タブで拒否を承認に変えると、**閉じても承認が保持** されます（以前は静かに破棄されていました）(v2.1.191)
- **サンドボックスのネットワーク権限がホストを記憶**：ネットワーク権限ダイアログで 'Yes' で許可したホストを **セッション中は記憶** し、接続のたびに再確認しません (v2.1.191)
- **管理設定の強制リフレッシュ修正**：`forceRemoteSettingsRefresh` が **MDM・ファイルポリシーで設定しても反映** されるよう修正され、取得要求に `Cache-Control: no-cache` を送って **プロキシが古い応答を返すこと** を防ぎます (v2.1.191)
- **ターミナル UI 修正多数**：起動 splash アートが **80×24 の macOS Terminal** をはみ出す問題、Windows Terminal で `/login` URL が **折り返しで切れる** 問題、ssh/tmux 上の Ghostty で **Cmd+click** が効かない問題、ストリーミング中に **スクロールが最下部へ飛ぶ** 問題などが修正されました (v2.1.191)

## おすすめコラム＆読み物

- **「就職とソフトウェアは終わった」**：10年目（元 Blizzard）のエンジニアが、**AI が増幅した採用市場の崩壊** を正面から扱った記事です。鋭い診断は **「ルールを守る応募者ほど不利になる構造」** — コーディング評価でこっそり AI を使う側が有利になり、企業はキーワードベースの履歴書フィルターやドキュメントアクセスを塞ぐ AI 監視試験で対応しますが、肝心のスマホ経由の AI 使用は止められません。より重いのは **ジュニアのはしごをまるごと外す** 流れ — 「AI がジュニアの役割をなくす」という期待のなか、新卒の入り口が閉じます。著者は AI コーディングの導入を *職人・テスター・エンジニアの職業的尊厳を放棄すること* と捉え、それに屈しないと宣言します — 6/21 の 'AI は能力を蝕むか'、6/23 の 'The Untrainable' と同じ位置で、*道具が仕事を圧縮するとき、人と職業に何が残るか* を最も生々しく問う記事です。[urflow](https://urflow.bearblog.dev/jobs-and-software-is-fucked/)
- **「AI 時代のプログラマー：コード生成から非決定性の制御へ」**：AI がコード生成を自動化するほど、開発者の核心的役割が **設計・検証・制御へ移る** という分析です。中心命題が明快です — **プログラマーの本質はコードを多く打つことではなく、曖昧な要求のディテールを埋め、それを再利用可能な形に抽象化することにある。** だから開発者は *置き換えられる* のではなく **システム設計者・ハーネスエンジニア** へ進化し、**テスト・コンパイラ・リンター・承認ゲート** といった決定論的な検証装置で AI の非決定性を御し、問題定義・アーキテクチャ・リスク管理という戦略的ディテールを握ります。6/21 の 'surfingcomplexity — 統合・検証は人が'、6/23 の 'Who Will Price Intelligence' と重ね、*何を自動化し、何を自分で握るか* を **ハーネスエンジニアリング** の言葉で整理した記事です。[velog (韓国語)](https://velog.io/@teo/we-programmer)

## 注目プロジェクト＆ツール

- **ponytail — AI エージェントを「最も怠惰なシニア開発者」のように考えさせる**：AI コーディングエージェントに **「最良のコードは、そもそも書かないコード」** という怠惰なシニアの考え方を注入する OSS です。コードを生成する前に **6段階のチェック** を強制します — *本当に必要か → 標準ライブラリで済むか → プラットフォームのネイティブ機能があるか → 既存の依存で済むか → 1行で終わるか* といった具合に、セキュリティ・データ整合性・アクセシビリティは妥協せず **過剰設計（over-engineering）を削ぎ落とします。** `/ponytail [lite|full|ultra|off]` で厳格度を調整し、`/ponytail-review` で diff の過剰設計を指摘し、`/ponytail-audit` でリポジトリ全体を分析します。一例で **コード54%減・安全性100%維持・コスト約20%減・速度約27%向上** を報告し、**Claude Code・Cursor を含む14エージェント** と互換、MIT ライセンスです — 上の '非決定性の制御' コラムの *抽象化は慎重に* という原則をエージェントのガードレールへ移した道具です。[GitHub](https://github.com/DietrichGebert/ponytail)
- **Monet — AI コーディングエージェントのための「ローカルメモリ」**：**「新しいセッションを開くたびに白紙になる」** というエージェントの持病を解こうとするメモリシステムです。3つの原則で動きます — **①自律的な記録**（エージェントが決定・パターンを自分でノート）、**②関連度ランキング検索**（よく参照されるメモリが先に浮上）、**③漸進的な改善**（蓄積するほどプロジェクトに特化）。単純な markdown ファイルから **構造化されたメモリインデックス** へ進化し、**100% ローカル**（データが端末を出ない）・**SQLite 保存（`~/.monet`）**・**無料**・**開かれた MCP インターフェース** で **Claude・Cursor** など MCP 対応エージェントと接続します — 6/23 で扱った Agent-Blackbox が *エージェントがどこでトークンを漏らすか* を測ったとすれば、Monet は *エージェントに何を記憶させるか* を扱う対の存在です。[Monet](https://monet.team-monet.com)
