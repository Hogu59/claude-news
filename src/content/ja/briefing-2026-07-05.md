## 主要な新機能と実践活用

### 管理コンソールに Claude Code 専用の Value・Usage タブ — コードアシスタントの価値を数字で (7/2)

**7/2、Anthropic が管理コンソールに Claude Code 専用の分析タブを2つ追加**しました。これまでのガバナンス機能が *誰が何を使えるか*(モデル・effort entitlements、spend caps)を制御するものだったのに対し、今回のタブは *その利用が実際にどれだけの価値を生むか*を管理者が見られるようにするピースです。

- **Usage タブ**: **アクティブ開発者数・セッション数・組織全体の上位コマンド(top commands)**を示し、**毎日更新**されます — 導入が実利用につながったか、どのコマンド・ワークフローが定着したかを見ます。
- **Value タブ**: **生産性向上(productivity lift)・コミットあたりコスト(cost per commit)・年間価値(annual value)**を推定します。要は **すべての算式(formula)をタブ内にそのまま見せ、入力値を組織の前提に合わせて調整**できるようにした点です — *ブラックボックスの ROI 数値*ではなく、検証・修正できる計算として提示します。
- **SCIM グループフィルタ**: IT が既に管理する SCIM グループ単位で分解されるので、利用状況が **既存の組織図そのまま**に分かれます。
- **自然言語 Analytics chat**: *今月使用量が2倍になったチームは?*、*シートあたり価値が最も高いのは?* といった質問を平文で投げると、**チャートを生成しエクスポート・共有**できます。

要点は **「組織に配備した Claude Code が値を出しているかを、管理者が自ら検証できる形で測る」**ことです — 6/29 の **Org default**、7/1 の **モデル・effort entitlements**、7/3 の **apps gateway・spend limits** と続いたガバナンスの最後のピース(*制御*から *価値測定*へ)です。Enterprise の管理者なら、Value タブの既定算式(特にコミットあたりコスト・生産性向上の前提)を自組織の実測値で一度校正しておくと、更新交渉や予算防衛の根拠に使いやすくなります。[Anthropic](https://claude.com/blog/giving-admins-more-visibility-and-control-over-claude-usage-and-spend)

## 開発者ワークフローティップス

### コンテキストを画像で入れてトークンコストを削る — pxpipe

**pxpipe** は大きなテキストコンテキストを **PNG 画像にレンダリングして API コストを下げるローカルプロキシ**です。着眼点が巧妙で、**画像トークンは文字数ではなくピクセルサイズで課金**されるため、密度の高いコード・JSON はテキストより画像で表現した方がトークンあたり効率的になり得ます。作者は **Fable で59〜70%のコスト削減**を報告しつつ、新規の算術・コード課題での正確性は維持されたとしています。

```text
# 適用判断の目安(pxpipe の報告ベース)
- 向いている: 大量のコード・JSON・ドキュメントなど「読んで理解」が目的の密なコンテキスト
- 注意: 正確な文字列再現が重要な場合(特に hex 識別子)→ 精度が落ちる
        こうした値は画像化せずテキストのまま保つ
```

このアイデアは **「モデルに必要なのは正確なバイトではなく理解だ」**という前提をコスト最適化に使います — 逐語引用・ハッシュ・ID 照合のようにバイト単位の正確さが要る作業には向かないので、**コンテキストを「理解用」と「正確再現用」に分け**、前者だけ画像化するのが安全です。昨日扱った **Fable 5 の上限・クレジット移行(7/7〜)**と併せて読むと、上限が厳しくなる局面で試す価値のある実験カードです(報告値なので、パイプライン導入前に代表的なタスクで自分で検証を)。[GeekNews](https://news.hada.io/topic?id=31127) · [GitHub](https://github.com/teamchong/pxpipe)

### 長いセッションではなく「コンテキスト0」リセットを繰り返す — retry-now パターン

**retry-now** はコードベースを **各イテレーションごとにコンテキスト0の新セッションで見直させる**自律ループエージェントです。作者の洞察が核心で、長いセッションのエージェントは *過去の決定を守ったり一つの方向に囚われたりする*傾向があるが、**毎回まったく新しい文脈でコードを読み直させれば**その慣性が消える、というものです。headless エージェントセッションを繰り返し起動して **分析 → 改善 → 検証 → 文書化**を回し、Claude Code・Codex CLI・OpenCode をサポートします。

反直感的な要点は **「コンテキストを長く保持することが常に得とは限らない」**ことです — 昨日のブリーフィングの *コンテキストが埋まるほどモデルの自己判断が鈍る*(だから外部の検証ハーネスが要る)という観察と正確につながります。ツールをそのまま使わなくても、**長い無人作業でセッションを丸ごと続けるのではなく、安定したチェックポイントごとにコンテキストをリセットして読み直させる**パターンは参考になります — 下の推薦コラム「ループエンジニアリング」が言う *ハーネスは繰り返すほど良くなる*と同じ場所です。[GeekNews](https://news.hada.io/topic?id=31111) · [GitHub](https://github.com/owjs3901/retry-now)

## おすすめコラム＆読み物

- **「ループエンジニアリングの美学」**: **エージェントの真の潜在力は、モデルそのものではなくモデルの周囲に積んだ「ループ」にある**という見方を4層で整理した記事です(LangChain)。① **agent loop** — モデルがツールを繰り返し呼んでタスクを完了します。② **verification loop** — 産出物をルーブリックに照らし、失敗ならフィードバックを戻して再試行します。③ **event-driven loop** — webhook・スケジュールなど外部トリガーでエージェントをより大きなシステムに組み込みます。④ **hill-climbing loop** — プロダクションのトレースを分析エージェントが読み、**ハーネス設定そのものを自動改善**します。要点はこれらのループが **入れ子**になり、外側のループが内側を次第に良くすることです — 6/29 の「ハーネスはモデルより重要」、7/4 の「AI との冒険(答えはより強い検証ハーネス)」の結論に *具体的な設計図*を添えた場所です。Claude Code の上に自前のエージェントパイプラインを載せるチームなら、上の retry-now・検証ハーネスの議論と併せて読む地図になります。[GeekNews](https://news.hada.io/topic?id=31106) · [原文](https://www.langchain.com/blog/the-art-of-loop-engineering)
- **「最新水準の LLM をローカルで動かす実践ガイド」**: フロンティア級モデルを **自前のハードウェア**で動かす際の構成とコストを実測でまとめたリポジトリです。2つのティアが鮮明です — 約 **2,000ドルのデュアル RTX 3090** で **Qwen3.6-27B** を動かす入門構成、そして約 **40,000ドルのクアッド RTX 6000 Pro(合計 384GB VRAM)**で **near-Claude-Opus** 水準を狙う上位構成です。後者は **c-payne PCIe Gen4 スイッチ**で GPU-GPU 直結を加え、**単方向 27.5GB/s・双方向 50.4GB/s の P2P 帯域**を出します。HN の議論は *量子化が複雑な課題で品質を落とし、ベンチスコアと実使用の体感が大きく異なる*現実を率直に指摘します — 6/30 の「Qwen 3.6 27B はローカル開発の最適点」が *モデル品質*を論じたのに対し、この記事は *それを実際に動かすハードウェア・コスト*を埋める続編です。プライバシー・オフライン・規制ドメインで自前推論を検討中ならコスト感をつかむのに有用です。[GeekNews](https://news.hada.io/topic?id=31121) · [GitHub](https://github.com/jamesob/local-llm)

## 注目プロジェクト＆ツール

- **Cluedoc — コーディングエージェントが作ったコードベースを、人が理解できるよう文書化する Agent Skill**: コーディングエージェントが吐き出したコードを **人が読めるドキュメント**にする **Agent Skill** です。システムを **feature ツリー**に整理し、**feature ごとに markdown ドキュメント1つ**を生成、別途のサーバー・DB なしで **ドキュメントをコードベース内に保存**してエージェントワークフロー内で最新状態に保ちます。7/3 の Godot・Webernetes が投げた *AI がコードを吐く時代の本当の問題は品質ではなく、誰が理解・保守するか*という問いへの **ツール的な答え**です — レビュー・保守の前提である *理解可能性*をドキュメント層で自動的に支えようとする試みです。Agent Skill として配布されるので、スタック型スキル(v2.1.199)とも組み合わせやすいです。[GeekNews](https://news.hada.io/topic?id=31133) · [Cluedoc](https://keunwoopark.github.io/cluedoc/)
- **llm-wiki-newsroom — 「マルチエージェントはトークンを食うだけ」への新聞編集局式の解**: *マルチエージェントはトークンを多く使い、コンテキストをよく取りこぼす*という問題意識から生まれたフレームワークで、自律エージェントを並列に回す代わりに **新聞編集局構造**を採用します — **判断は唯一ひとつの「desk 編集者」LLM だけ**が行い、他の役割は決められた作業(執筆・Python によるリント検証・オーケストレーション)だけを実行します。ソースから **相互リンクされた markdown ドキュメント**を積み、エンティティ・概念を抽出して **総合概要・矛盾(contradiction)ページに合成**し、すべて git に保存してローカルで処理するため **Claude Code アクセス以外に API キーは不要**です。今日のデイリーブリーフィングそのものとメタ的に重なる構造であり、*判断ノードは1つに絞り、残りは決定論的な実行に任せよ*という設計の教訓が刺さります — 上の「ループエンジニアリング」の検証ループ、retry-now のコンテキストリセットと同じ問題(マルチエージェントの信頼性・コスト)を *構造*で解いた事例です。[GeekNews](https://news.hada.io/topic?id=31140) · [GitHub](https://github.com/alfadur7/llm-wiki-newsroom)
