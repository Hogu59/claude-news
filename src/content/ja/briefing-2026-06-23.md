## 主要な新機能と実践活用

### `claude mcp login / logout <name>` — `/mcp` メニューを開かず CLI から MCP を認証する (v2.1.186)

今回のリリースのヘッドラインです。**MCP サーバーの認証を CLI コマンド1行で**処理できるようになりました — インタラクティブな `/mcp` メニューを開く必要はありません。

```bash
# 特定の MCP サーバーを認証 / ログアウト（対話メニューなし）
claude mcp login github
claude mcp logout github

# SSH セッションなどブラウザを開けない環境：stdin リダイレクトで完了
claude mcp login github --no-browser
```

実務上の意味は明確です — これまで OAuth が必要な MCP サーバーをつなぐには、**インタラクティブなセッションで `/mcp` メニューを開き**、クリックで認証する必要があり、**SSH・ヘッドレス・CI のオンボーディング**とは相性が良くありませんでした。`--no-browser` オプションは、**ブラウザを起動できないリモートシェル**でも認証フローを stdin で受けて完了させます。6/21 の **WIF（静的キーの排除）**、6/20 の **マネージド MCP connector** が *組織単位* の認証を扱ったのに対し、今回は **個人開発者がリモートサーバーで手動で MCP をつなぐ際の摩擦**を減らす一手です。加えて `claude mcp get`/`remove` は、**タイポ時に最も近い設定済みサーバー名を提案**し、長いリストは省略表示するようになりました。 [GitHub v2.1.186](https://github.com/anthropics/claude-code/releases/tag/v2.1.186)

### `!` bash コマンドの出力に Claude が「自動で応答」する — 不要なら無効化する方法 (v2.1.186)

小さいながら体感できる挙動変更です。プロンプトで `!` を使って直接 bash コマンドを実行したとき、その**出力に対して Claude が自動で応答**するようになりました — 従来は、コマンド結果を**コンテキストに追加するだけ**で、モデルはすぐには反応しませんでした。

```bash
# 出力にモデルが直接反応する（例：失敗原因の分析・次ステップの提案）
!npm test

# 従来の「コンテキスト追加のみ・応答しない」に戻すには settings.json:
# { "respondToBashCommands": false }
```

クイックなコマンド結果をその場でモデルに解釈させたいフローには便利ですが、**ただ出力を確認したいだけ**の場合は意図しない応答（とトークン消費）が生じ得ます。`!` でコマンドを頻繁に回す方は、新しいデフォルトが自分のワークフローに合うか確認し、合わなければ **`respondToBashCommands: false` で旧挙動を復元**してください。 [GitHub v2.1.186](https://github.com/anthropics/claude-code/releases/tag/v2.1.186)

## 開発者ワークフローティップス

### 無人セッションのリトライガードが変わった — `MAX_RETRIES` は15にキャップ、無人実行には `RETRY_WATCHDOG`

CI・cron で Claude Code を無人実行している方は、今回のリリースのこの1行を押さえてください — **`CLAUDE_CODE_MAX_RETRIES` が最大15にキャップ**されました。大きな値を入れて「ほぼ無限にリトライ」させていたパターンは、もう通用しません。リリースノートは、**無人（unattended）セッションには代わりに `CLAUDE_CODE_RETRY_WATCHDOG` を使う**よう明記しています。

ポイントは **自動化のリトライ戦略を見直せ**というシグナルです。短いバックエンドの揺らぎ（今週もノイズがありました）に備えてリトライ回数だけ増やしていたパイプラインは、その値が15で切られるため挙動が変わり得ます。無人で長時間ジョブを回すなら、**リトライ上限を明示的に点検**し、無人シナリオで推奨される watchdog メカニズムへの移行を検討してください（正確な設定キー・値は公式の settings ドキュメントに従ってください）。 [GitHub v2.1.186](https://github.com/anthropics/claude-code/releases/tag/v2.1.186)

### バックグラウンドのサブエージェントの権限モデルが変わった — 自動拒否 → メインセッションに表示

これまで、バックグラウンドのサブエージェントは権限が必要なツールに遭遇すると**静かに自動拒否**され、無人で回していたバックグラウンド作業が**理由もなく止まったように**見えることがありました。v2.1.186 からは、バックグラウンドのサブエージェントの権限要求が**メインセッションにプロンプトとして表示**されます — ダイアログには**どのエージェントが何を要求しているか**が示され、**Esc はそのツール1つだけを拒否**します。

これと対になるセキュリティ修正があります — `Agent(type)` の **deny ルール**と `Agent(x,y)` の**許可タイプ制限**が、これまで抜けていた**名前付き（named）サブエージェントの spawn にも適用**されるようになりました。6/19 で扱った「サブエージェントに最小権限を与え、git にコミットせよ」の延長線です — 権限ルールを `.claude/` に固定していれば、そのルールがバックグラウンド・named の経路まで一貫して適用されます。バックグラウンド/チームのワークフローを回すなら、「プロンプトはメインに出る」という前提で、**無人自動化で誰がそのプロンプトに答えるか**を再設計しておきましょう（完全無人なら、必要なツールを allow ルールであらかじめ開けておく形）。 [GitHub v2.1.186](https://github.com/anthropics/claude-code/releases/tag/v2.1.186)

## セキュリティ・制限事項

### Claude の本人確認、7/8 発効 — ただし「全員顔スキャン」ではなく選択的・異議申し立て型 (6/22 発表)

Anthropic が **Claude の本人確認（identity verification）**を 6/22 に告知し、**7/8 発効**を発表しました。一部メディアは「ID がなければアクセス不可」「全員が顔スキャン」と報じましたが、**公式ヘルプセンターの文書を読むと全員義務ではありません。**

- **選択的・機能単位**：公式ドキュメントは、本人確認が **「いくつかのユースケースに限って導入される」**とし、ユーザーは **「特定の機能にアクセスする際に確認プロンプトを見ることがある」**と説明しています。全ユーザーが 7/8 に一斉認証する、という意味ではありません。
- **異議申し立て（appeal）プロセスの一部**：TechCrunch によると、Anthropic の Thariq Shihipar 氏はこれを **「異議申し立てプロセスの更新」**と説明しました — 即時の永久 BAN ではなく、**異常で flag されたアカウントが本人確認でアクセスを復旧**する道です（BAN されたアカウントは `claude.ai/restricted` で申請フォームを提出 → Safeguards チームが調査）。
- **収集・処理**：**政府発行の写真付き ID** と（場合により）**ライブセルフィー**を受け取り、処理は外部の KYC 事業者 **Persona Identities** が担います（ID・セルフィーは Persona のサーバーに保存）。Persona は **Peter Thiel の Founders Fund が出資**しており（Thiel は Anthropic の投資家でもあります）、**Discord が年齢確認に Persona を導入してユーザーの反発で撤回した前例**があるため、プライバシー上の懸念が指摘されています。
- **根拠**：年齢確認規制の遵守、不正防止、ToS の執行、セキュリティ問題の調査などが理由として挙げられています。Anthropic は **「確認データは本人確認の目的以外には使用しない」**と明記しています。

開発者視点の核心は **範囲を正確に読むこと**です — 報道は消費者向け Claude（Free/Pro/Max）の文脈に偏っており、**API・Claude Code への適用は公式には確認されていません。** 誇張された「全員顔スキャン」の語りとは距離を置きつつ、**個人の購読アカウントで Claude Code を認証して使っている場合**、アカウントが flag されると復旧ステップで本人確認に遭遇し得る、という程度に理解すれば十分です。 [TechCrunch](https://techcrunch.com/2026/06/22/anthropic-says-claude-may-want-to-see-your-id/) · [Anthropic ヘルプセンター](https://support.claude.com/en/articles/14328960-identity-verification-on-claude)

## エコシステム＆プラグイン

### agent-connector — MCP サーバー/hook を「一度定義」すれば42の CLI に合わせて自動生成

複数のエージェント CLI に同じ MCP サーバーをつなぐたびに、**ツールごとのフォーマット（JSON・TOML・settings）で個別に設定**をコピペしていた摩擦をなくすオープンソースです。**connector を一度定義**すれば、**Claude・Codex・Cursor・Gemini など42以上の CLI ツール**向けの**ネイティブ設定ファイルを自動生成**します。

- **1つの定義 → 各 CLI のフォーマット**：同じ MCP サーバー/hook 定義をツールごとの形式にレンダリングし、**hook イベントモデルをホストプラットフォーム別に正規化**します。
- **運用機能**：プラグイン/拡張のパッケージング、マーケットプレイス式のインストールフロー、**ツール別のトークンテレメトリ**、Linux・macOS・Windows のクロスプラットフォームテスト、`--purge` によるクリーンな削除まで提供します。
- **効果**：既存の MCP サーバーをこの方式でポーティングした事例では、**デプロイコードが 20,322行 → 76行**に減り、対応 CLI は **15 → 42** に増えたとされています。

6/22 で扱った **opencodex**（Codex であらゆる LLM を使えるようにするプロキシ）が *モデル互換性* を狙ったのに対し、agent-connector はその一段下で **「エージェント設定のデプロイ」を単一ソースに統合**する対の存在です — MCP サーバーを作り、**複数のエージェントに同時にデプロイ**したい人に実用的です。 [GitHub](https://github.com/ken-jo/agent-connector)

## おすすめコラム＆読み物

- **「The Untrainable（訓練できないもの）」 (Sara Normous)**：AI モデルが *測定可能な* 領域でコモディティ化するほど、持続的な競争価値は **「訓練できないもの」**へ移る、という分析です — **私的な正解**（企業ごとに異なるデータ）、**検証コストの高い仕事**、**規制責任・ライセンス**、**顧客との関係や組織の信頼**といった領域です。最も鋭い論拠は、AI エージェントが **コーディングのベンチマークを13%（2024）から18か月で80%超**へ押し上げたのに、**実際にデプロイされたコードは約30%しか増えていない**という点。つまり **ベンチマーク性能 ≠ ビジネス価値**であり、ボトルネックは *知性* ではなく **権限と説明責任（permission and accountability）**だ、という診断です。6/18 の「Addy Osmani — コード4倍なのに生産性は+12%」、6/21 の「AI は私たちの能力を損なうか」と同じ位置 — 「エージェントが *できない* こと」こそ、人と組織が価値を置く場所だという話です。 [The Untrainable](https://saranormous.substack.com/p/the-untrainable)
- **「誰が知性に値段をつけるのか（Who Will Price Intelligence?）」 (Jaya Gupta)**：AI が **ソフトウェアとは根本的に異なる理由**を、価格と市場構造から読み解きます。SaaS では脅威は競合ですが、AI では **性能・コスト・レイテンシ・デプロイ・規制・人材・地政学**といった変数が **それぞれ独立に、互いに予測不能に**動くため、**防御可能な固定ポジションが存在しない**というのです。だから仕事は モデル→オープンウェイト→デバイス と絶えず **「レイヤー移動（layer migration）」**し、企業は防御的なプロダクトを積み上げるより、**ロング/ショートを計算する「トレーディングブック」のように**運用すべきだと見ます。6/20 の「Qwen はより安い Opus ではなく別のツール」、6/22 の「ClickHouse の10年」と重なり — モデルの可用性や価格が揺れる今、**「何を固定し、何を差し替えるか」**をポートフォリオのように扱え、という視点です。 [Jaya Gupta (X)](https://x.com/JayaGup10/status/2065965053334216718)

## 注目プロジェクト＆ツール

- **Agent-Blackbox — Claude Code/OpenCode の「実際の動作」を記録してコンテキストの無駄を診断**：エージェントの *要約* に頼らず、**ファイル読み込み・bash コマンド・検索・トークン使用**といった実イベントを直接記録し、「エージェントがどこでコンテキストを漏らしているか」を可視化するモニタリングツールです。**同じファイルを繰り返し読む**、**大きなコマンド出力がコンテキストを食う**、**失敗したコマンドを根本原因の修正なしに繰り返す**といったパターンを指摘し、セッションごとに **どのファイルに触れ、どこでトークンが浪費されたか**を可視化します。**`npx` でインストール不要**で動き、**API キーなしでローカル**に動作。ある事例では **939k → 521k トークン**に削減し、効率スコアが 80→99 に上がったとのこと — 6/15 のクレジット分離以降、「自分のエージェントがどこでトークンを燃やしているか」を測りたい人に実用的です（長時間・マルチエージェント作業では特に）。 [GitHub](https://github.com/TaewoooPark/Agent-Blackbox)
- **Repolis — GitHub の公開リポジトリを「3D 都市」として歩く**：GitHub ユーザーの公開リポジトリを **インタラクティブな3D都市**に変えるプロジェクトです。リポジトリ1つが **建物1つ**になり、**活動量・トラフィックに応じて建物の高さやディテール**が変わります。肝は **AI「タクシー」インターフェース** — `"AI agent"`・`"最も人気のある repo"` のように自然言語で尋ねると、該当リポジトリ（建物）まで連れて行ってくれます。GitHub の **6個の pin 制限**を越えて **すべての公開プロジェクトを一望**でき、埋もれていた古い・アーカイブ済みのリポジトリまでポートフォリオの文脈に引き上げます。「AI エージェントがコードベースを探索する方式」を可視化に落とし込んだ発想が新鮮な、開発者ポートフォリオの代替表現です。 [GitHub](https://github.com/hyeonsangjeon/Repolis)
