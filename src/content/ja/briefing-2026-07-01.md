## 主要な新機能と実践活用

### Claude Sonnet 5 — Opus に近いのにずっと安い、エージェント向けデフォルトモデル（6/30）

今週最大の発表です。**6/30、Anthropic が Claude Sonnet 5 を発表**し、同日 **v2.1.197 で Claude Code のデフォルトモデル** になりました。位置づけは明確です — **史上最もエージェンティックな Sonnet** として、計画を立て、ブラウザやターミナルなどのツールを使い、長時間自律的に動くよう設計されています。少し前まで大きく高価なモデルを要した作業を、Sonnet の価格帯で処理するというのが核心メッセージです。

- **コンテキストと推論**: **ネイティブ1Mトークンのコンテキスト** をサポートし、**adaptive thinking が常時ON** です。特に **effort が API・Claude Code で high デフォルト** に設定され、明示的な指示なしでも多段のソフトウェア作業・デバッグ・PR 処理を最後まで押し切ります。**ナレッジカットオフは2026年1月** です。
- **性能（公式発表ベース）**: 発表は精密なスコア表ではなく **コスト性能曲線** で説明します — エージェンティック検索・コンピュータ操作の評価（**BrowseComp・OSWorld-Verified**）で **Opus 4.8 に近い性能をより低いコストで** 出すというものです。*以前の Sonnet が途中で止まっていた複雑なタスクを完遂する* という初期テスターの声も添えられています。
- **性能（サードパーティ集計ベース）**: 報道・分析メディアの集計では **SWE-bench Pro 約63.2パーセント**（Sonnet 4.6 約58.1パーセント、Opus 4.8 約69.2パーセント）、知識労働ベンチ **GDPval-AA v2 約1,618**（Opus 4.8 約1,615 とほぼ同等）と伝えられます。ただしこれらの精密な数値は **Anthropic の公式発表には明記されていない** ため、参考指標として読んでください。
- **価格と提供**: 導入価格は **8/31まで100万トークンあたり2ドル（入力）/10ドル（出力）**、以降 **3ドル/15ドル** です。**Free・Pro のデフォルトモデル** となり、**Claude Code・Claude API はもちろん Cursor・VS Code・GitHub Copilot にも初日から** 搭載されました。

開発者にとっての核心は **デフォルトの選択肢が変わった** ことです — *エージェントを長時間動かすには高価な Opus を使うべきか* という悩みの重心が、**Opus に近い品質をずっと安く** 出す Sonnet 5 側へ移りました。下のワークフローで *effort とコスト*、そして *いつ依然 Opus 4.8 を使うか* を続けて扱います。[Introducing Claude Sonnet 5](https://www.anthropic.com/news/claude-sonnet-5) · [TechCrunch](https://techcrunch.com/2026/06/30/anthropic-launches-claude-sonnet-5-as-a-cheaper-way-to-run-agents/)

### Claude Science — 研究者向け AI ワークベンチも同日ローンチ（6/30）

Sonnet 5 の陰に隠れましたが、同じ6/30に Anthropic は **Claude Science** も投入しました — **研究者向けの AI ワークベンチ** です。研究者が最もよく使う **ツールやパッケージを統合** し、**監査可能（auditable）な成果物** を生成し、**計算リソースへ柔軟にアクセス** できるようにするカスタマイズ可能なアプリです。

核心は **再現性・監査可能性を備えた科学ワークフロー** を狙う点です — LLM がコードを大量に吐く時代に、*成果物が検証可能で追跡可能であるべき* という要求は研究ドメインで特に重くなります。6/27の SongRyeon（判断と検証済みの事実の分離）、本日下の Klorn（LLM はスコアのみ・決定はルール）と同じ筋で — *AI の出力を、もっともらしい物語ではなく監査可能な根拠へ戻す* 流れが、今回は Anthropic の正式プロダクトとして出た事例です。[Anthropic Newsroom](https://www.anthropic.com/news)

## 開発者ワークフローティップス

### Sonnet 5 は effort が high デフォルト — コストは「トークン単価」でなく「タスク単位」で測り直す

Sonnet 5 へ移行してすぐ注意すべき落とし穴です。Sonnet 5 は **effort（推論強度）が API・Claude Code で high デフォルト** です。つまり **トークン単価は Sonnet 4.6 や Opus より安くても**、1タスクが **より多くの推論トークンを使う** ため、*タスク単位の総コスト* は単価だけで期待したほど下がらない可能性があります。

```bash
# Sonnet 5 は effort が high デフォルト。多くのエージェンティックコーディングは medium が均衡点
claude --model claude-sonnet-5 --effort medium

# 大量・低レイテンシのワークロード（チャット・非コーディング）は low へ
claude --effort low
# 設定にデフォルト effort を固定するか、リクエスト単位で --effort で上書き
```

要点は **安い単価と安いタスクを混同しない** ことです — max/high effort では同じプロンプトが low 比で **10倍以上のトークン** を使うこともあります。多くのエージェンティックコーディングやツール中心のワークフローは **medium が速度・コスト・品質の均衡点** で、本当に難しい推論にだけ high を残すのが実利的です。Sonnet 5 へパイプラインを移したなら、**更新直後に実タスク数件のトークン消費を測って** からデフォルト effort を決めてください。[effort ドキュメント](https://platform.claude.com/docs/en/build-with-claude/effort) · [モデル設定](https://code.claude.com/docs/en/model-config)

### それでも「最難関タスク」は Opus 4.8 へ — デフォルトは Sonnet 5、頂点は Opus に分ける

2つ目は **モデル選択の新しい地図** です。Sonnet 5 がデフォルトになったからといって、すべてを Sonnet に寄せる必要はありません。公式発表もサードパーティ数値も同じ方向を指します — **Sonnet 5 は Opus 4.8 に「近い」が、まだ上回ってはいません**（サードパーティ集計で SWE-bench Pro 約6ポイント差）。つまり *大半の日常作業* は安価な Sonnet 5 で処理し、**最も厄介なアーキテクチャ判断・長いデバッグ・精度が勝負を分けるタスク** は依然 Opus 4.8 へ上げる **2段構成** が、コスト・品質の両面で有利です。

```jsonc
// デフォルトは Sonnet 5、難しいタスクだけ Opus 4.8 へエスカレーション
{
  "model": "claude-sonnet-5",
  "fallbackModel": "claude-opus-4-8"  // 正確なキー/値は公式 model-config ドキュメントに従ってください
}
```

6/28の **作業者と採点者の分離**（検証は別モデルに）、6/29の **ハーネスはモデルより重要** と同じ筋です — *どのモデルをどこで使うか* を設計で分けておけば、安いデフォルトで大半を捌きつつ、頂点品質が必要な瞬間だけ上位モデルへ正確に上げられます。[Introducing Claude Sonnet 5](https://www.anthropic.com/news/claude-sonnet-5)

## セキュリティ・制限事項

### Fable 5 の復帰予想は7/1もハズレた — だが Sonnet 5 は「輸出規制の外」なので外国籍の開発者も使える

輸出規制をめぐる **本日の逆転** です。複数メディアが **7/1 頃の Fable 5 復帰** を予想しましたが、**予測はまたハズレました** — 7/1時点で **Fable 5 は依然として全ユーザー（消費者・API・Claude Code・海外サブスク）にオフライン** で、公式の復帰日はありません。6/27の Lutnick 書簡による部分緩和は **Mythos 5 を Annex A パートナーにのみ** 戻しただけで、**Fable 5 はその対象外** です。

- **本日の本当のニュースは Sonnet 5 側です。** Fable 5・Mythos 5 が輸出規制でブロックされている間、**Sonnet 5 は輸出規制の対象外のまま広く出荷** されました — Free・Pro のデフォルトモデルであり、Claude Code・API で **国籍制限なく** 使えます。つまり **外国籍の開発者（韓国・日本など）にも、新しいフロンティア級モデルが1つ開かれました。**
- **実務上の含意**: これまでが *ブロックされた Fable 5 を待ちつつ Opus 4.8 でしのぐ* だったなら、いまは **より安く新しい Sonnet 5 がフォールバック経路に加わりました。** `fallbackModel`・`availableModels` に **Sonnet 5 と Opus 4.8 を併記** しておけば、モデルの可用性が地政学に揺れても *アクセス可能な* 最新モデルで自動化を続けられます。

核心の教訓は — **輸出規制はすべての新モデルをブロックするわけではない** という点です。規制対象（Fable 5・Mythos 5）と一般提供（Sonnet 5）を **区別して読み**、アクセス可能な最新モデルをフォールバック経路に反映してください。[Anthropic 声明（Fable/Mythos）](https://www.anthropic.com/news/fable-mythos-access) · [explainx](https://explainx.ai/blog/when-will-fable-5-be-available-again-2026)

## 知っておくと便利な小さな変更点

Sonnet 5 ローンチで上記に収まらなかった実用的なディテールです。

- **ナレッジカットオフ**: Sonnet 5 の学習ナレッジカットオフは **2026年1月** です

- **最小バージョン**: Claude Code で Sonnet 5 をデフォルトで使うには **v2.1.197 以上** への更新が必要です

## おすすめコラム＆読み物

- **「CUDA カーネルを実行すると内部で起こること」**: 単純なベクトル加算の CUDA プログラム1つが、**コンパイルから GPU 実行・完了まで** に辿る全過程を細かく解剖する記事です。`nvcc` が host/device コードを分離して **PTX（仮想 ISA）と SASS（アーキテクチャ別命令）** を生成し、launch stub がカーネル呼び出しを `libcuda.so.1` のドライバ命令へ変え、GPU は **pushbuffer・GPFIFO キュー** で作業を受け、**メモリマップされた doorbell レジスタ** で通知される、という具合に層を掘り下げます。圧巻は結論 — このカーネルは **算術強度が低く** 演算ではなく **メモリに縛られ（memory-bound）**、RTX 4090 の **DRAM 最大帯域の79.65パーセント** で回るという実測です。*AI が GPU を安く回す時代* こそ、**その下で実際に何が起きているか** を知ることが性能の限界を読む目になると示します。[Fergus Finn](https://fergusfinn.com/blog/what-happens-when-you-run-a-gpu-kernel/)
- **「演算コストを反映した LTV（Compute-adjusted LTV）の計算法」**: 本日の Sonnet 5 の *コストの物語* と正確に共鳴する記事です。核心の診断は — **従来の SaaS の LTV 指標が AI 製品では崩れる** こと。同じ月200ドルを払う2顧客でも、一方が推論コスト110ドル、他方が15ドルなら *平均マージン* は嘘になります。処方は **フルバーデン（fully-burdened）の AI 原価**（推論・インフラ・サポート・CS）を顧客単位の収益性に織り込み、*どのセグメントが実際に利益になるか* を可視化することです。根拠の数値が鋭い — **推論コストは AI B2B 売上の平均約23パーセント**、使用量の差は **最大319倍** に達します。著者は *推論コストが売上の10パーセントを超え、セグメント別の使用量の差が大きいとき* にこの指標が特に重要だと釘を刺します — エージェントを実際に動かすチームなら、**単価引き下げと同じくらい「誰がどれだけ使うか」** を見るべきという実戦フレームです。[The SaaS CFO](https://www.thesaascfo.com/how-to-calculate-compute-adjusted-ltv/)
- **「月額プランが最善の選択でありうる」**: 業界の *年額プラン万能論* を正面から反論する記事です。核心は — 年額プランの優秀に見えるリテンション指標が、実は **遅延したチャーン（delayed churn）** を隠すこと。ユーザーは興味を失っても更新前まで数か月 *ダッシュボード上はアクティブ* に残り、製品の本当の問題を覆い隠します。対照的なデータが印象的です — 生産性アプリでは **サブスクの76.7パーセントが月額なのに、売上の90.7パーセントが月額から** 生まれます（月額サブスクの方が安定して支払い、再活性化の可能性も高いため）。月額が有利な5つの状況（PMF 前の学習期・信頼構築・柔軟性志向・短期利用・リテンションより売上最大化）を挙げ、**月額を「後回し」ではなく成長段階に合わせた意図的な設計** として置くよう勧めます — 上の演算コスト反映 LTV と重ねて読むと、*AI 製品の価格・課金設計* を考え直させる対です。[RevenueCat](https://www.revenuecat.com/blog/growth/monthly-subscriptions-when-to-offer/)

## 注目プロジェクト＆ツール

- **Klorn — メールを AI に任せるな、前に「ファイアウォール」を置け（オープンソース）**: よくある AI メール秘書と **正反対の発想** のオープンソース・メールフィルタです。AI に返信を書かせたり自動送信させたりする代わりに、**受信箱の前に「ファイアウォール」を立て**、届くメールを4段階に分類します — **SILENT**（記録するが非表示）・**QUEUE**（通知なしで表示）・**PUSH**（緊急通知）・**AUTO**（現在はスコア化のみ）。設計の要は **役割分離** です: LLM は **信頼度・送信者の信用・取り消し可能性・緊急性** の4軸の *スコアだけ* を付け、**実際の決定は決定論的（deterministic）ルール** が下します。これにより **送信・完全削除・転送といった取り消せない行為** を LLM の判断だけに委ねず、*監査可能性* を保ちます。6/27の SongRyeon（判断と検証済み事実の分離）、6/29のハーネス設計、本日の Claude Science（監査可能な成果物）と同じ筋で — **「LLM の判断はそのまま実行せず、取り消せない行為はルールが握る」** という原則を受信箱に移したツールです。[GitHub](https://github.com/k08200/klorn)
- **web_harvester — 「何が変わったか」を教える Web コレクター（Chrome 拡張 + Python CLI）**: 一度きりのクロールではなく、**起動しておくとサイトの変更点を時間軸で追跡** するツールです。同じ項目を **日付別に Google Sheets・CSV・Excel・SQL へ蓄積** し、*スナップショット1枚* ではなく *トレンド* を見せます。**Slack・Discord・メールでの変更通知**、ノーコードのフィールドマッピング（自動候補検出）、JSON-LD・DOM パースから最適な抽出法を診断する **recon エンジン** を備え、**Chrome MV3 拡張 + Python CLI** の2形態で使います。作者は **CAPTCHA・プロキシの回避をせず、公開データ・低頻度リクエストを志向** すると明言します（収集・フィールド検出・エクスポートは無料、自動スケジュール・変更通知は有料）。価格・在庫・公告のような *変わる値* を定期的に見張る必要がある人に実用的です。[Chrome Web Store](https://chromewebstore.google.com/detail/jaipjjloacjlabdjngemnlkccghdhcfk)
