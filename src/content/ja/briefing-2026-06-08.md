## セキュリティ・制限事項

### 6/15 Programmatic Usage Credits — 自動化の利用がサブスクリプションから分離されます（請求案内メールは本日 6/8 から発送と報道）

これまで複数のブリーフィングが「6/15 に施行」とだけ触れてきた **Programmatic Usage Credits** の輪郭がはっきりしてきました。5月14日に予告されたこの変更は **6月15日から**適用され、報道によれば対象サブスクライバーへの請求（opt-in）案内メールが **本日（6月8日）から**発送され始めます。新たな発表ではなく、「施行が目前で、請求の窓口が本日開く」という点が本日のポイントです。

要点は **「プログラマティック（非インタラクティブ）な利用をサブスクリプションの上限から切り離し、別枠の月次クレジットに移す」** ことです。

- **分離される（クレジットを消費する）利用**：`claude -p`（ヘッドレス）、**Claude Agent SDK**、**Claude Code GitHub Actions**、そして Agent SDK で認証するサードパーティエージェント（OpenClaw・Conductor・Zed など）。
- **影響なし（サブスクリプションのまま）の利用**：ターミナル・IDE の**インタラクティブな Claude Code**、Web・モバイルの Claude チャット、Claude Cowork。
- **月次クレジット額**：Pro **$20**、Max 5x **$100**、Max 20x **$200**（Team は席ごとに $20/$100）。**標準 API レートで消費**され、**繰り越し不可（non-rollover）**で、ユーザー間の共有・移転はできません。
- **使い切ったとき**：オーバーフロー課金を自分で有効にしていない限り、自動化リクエストは**そのまま停止**します（自動フォールバックなし）。

やるべきことは3つです — **(1)** メールが届いたら 6/15 までにクレジットを **claim** する（未請求のアカウントは 6/15 以降にプログラマティック利用がブロック／スロットルされる可能性があります）、**(2)** CI や無人パイプラインなど、**自動化がどこでトークンを使っているか**を把握する、**(3)** 利用量の大きい自動化は、予測可能な課金のために**専用 API キーへの分離**を検討する。これまでのワークフローティップスで扱った `MAX_THINKING_TOKENS=0`・サブエージェントのモデル固定・`OTEL_RESOURCE_ATTRIBUTES` ラベルといったコストガバナンスの仕掛けは、いまや「サブスクリプションの上限」ではなく**実際のクレジット残高**に直接効くという点で、より重要になりました。（正確な条件や未請求時の挙動は公式の案内に従ってください。）

## 開発者ワークフローティップス

### 探索は Plan mode と Explore サブエージェントに任せ、メインのコンテキストは実装用に空けておく

最もよくあるコンテキストの浪費は、**「読み込みや探索の副産物がメインの会話に積み上がること」** です。2つの仕掛けでこれを切り離せます。

- **Plan mode**：コードに手を付ける前に**探索と計画だけ**を行うモードに入り、見当違いの問題を解くのを防ぎます。計画が固まったら実行に切り替えます。
- **Explore サブエージェント**：読み取り専用（Write・Edit 不可）で **Haiku 上で動く**高速な探索エージェントです。「これらのファイルを読んで X を調べて」と投げれば、別のコンテキストでコードベースをさらい、**要約だけ**を返します。

探索の冗長な出力がメインのセッションを埋めないので、実装フェーズでモデルが見るべき情報がすっきり保たれます。

[Claude Code Docs — Best practices](https://code.claude.com/docs/en/best-practices)

### ちょっとした質問は `/btw` で — 会話履歴を汚さない

作業の途中で「あのフラグ何だっけ？」のような**使い捨ての質問**をそのまま投げると、その質疑がメインの会話に永久に残り、コンテキストを侵食します。`/btw` で尋ねると、答えは**閉じられるオーバーレイ**に表示され、**会話履歴には入りません**。

```bash
/btw このリポジトリのデフォルトのテストコマンドは何だっけ？
```

上の Plan mode・Explore と同じ筋の**コンテキスト衛生**ツールです — 作業に不要な情報はメインのセッションの外に置きます。（v2.1.163 以降、`/btw` の回答は「c」で生の Markdown のままコピーでき、別の場所に貼っても書式が保たれます。）

[Claude Code Docs — Best practices](https://code.claude.com/docs/en/best-practices)

## おすすめコラム＆読み物

- **「ハーネスエンジニアリング：エージェント優先の世界で Codex を活用する」 — OpenAI**：社内ベータ製品を、**人が手書きしたコード0行**を固定の制約として、Codex エージェントだけで構築した事例です。エンジニア3名が5か月で **1,500 の PR・約100万行**を処理し、仕事はコードを書くことから**環境の設計・意図の明示・フィードバックループの構築**へ移った、といいます。Claude Code の Dynamic Workflows が言う「タスクごとのハーネス」とまったく同じ転換を、競合が自らの言葉で証言する一篇です。GN+ 6ポイント。[OpenAI](https://openai.com/index/harness-engineering/)
- **「fork() + exec() を超えて」 — LWN**：Linux のプロセス生成を、従来の `fork()+exec()` の先へ最適化しようというカーネルの議論です。同じバイナリを繰り返し実行する際に情報をキャッシュする Li Chen の「spawn templates」（約2%の改善）と、`pidfd_config()` システムコールで空プロセスを作り本来の `posix_spawn()` を目指す Christian Brauner の代替案がぶつかります。エージェントが絶え間なくシェルコマンドを起動する時代に、**「プロセスを起動するコスト」そのもの**を見直させる低レイヤーの読み物です。GN+ 12ポイント・コメント3。[LWN](https://lwn.net/SubscriberLink/1076018/16f01bbbb8e0d1f0/)
- **「米国の新卒の失業率が、平均的な労働者を上回った」 — Randal Olson**：新卒の失業率（5.6%）が労働者全体（4.2%）を上回り、**過去最大の差**を記録したというデータ分析です。興味深いことにこの逆転は **2019年から始まって**おり、AI の普及やパンデミックよりも前で、学位の構造的な価値低下というより、リモートワークと **AI への露出度が高い職種**が要因として指摘されています。「AI が新卒の仕事を奪った」という単純な物語を、データで分解してみせる記事です。[Randal Olson](https://www.randalolson.com/2026/06/04/recent-grad-unemployment-flip/)

## 注目プロジェクト＆ツール

- **NeoGraph — LangGraph を C++17 に移植したエージェントオーケストレーションエンジン**：Python の LangGraph を C++17 で再実装し、性能を引き上げたエージェントオーケストレーションエンジンです。エージェントのグラフをコードのオブジェクトではなく **JSON データとして定義**し、無停止のホットスワップとマルチテナント運用に対応、ベンチマークでは **ノードあたりのエンジンオーバーヘッド 5.0µs**（LangGraph の 643µs・AutoGen の 3,127µs に対して）を出すとしています。「エージェントの実行系を再デプロイなしで差し替える」という発想が面白いところです（PyPI の `neograph-engine` でも提供）。GN+ Show。[GitHub](https://github.com/fox1245/NeoGraph)
- **Claude Code Done Overlay — 自分の番になると画面を覆う通知プラグイン**：Claude Code がターンを終えて入力待ちになると、画面を**黒いオーバーレイで埋め尽くし**、応答を待つ間によそ見してその瞬間を逃さないようにする小さなプラグインです。作者も使い方によっては煩わしいかもしれないと注釈していますが、「エージェントが自分の番を待っている瞬間」を逃すというよくある不便を真正面から狙った、ミニマルなツールです。GN+ Show。[GitHub](https://github.com/roeniss/claude-code-done-overlay)
