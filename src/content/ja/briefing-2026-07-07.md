## 主要な新機能と実践活用

### Dynamic workflow size — マルチエージェントワークフローに「規模ダイヤル」が付いた (v2.1.202)

v2.1.202 は **`/config` に Dynamic workflow size 設定**を追加しました — workflow が起動する**エージェント数を small/medium/large で制御**します。これまでマルチエージェントワークフローの規模はスクリプトやプロンプト次第でしたが、*同じワークフローをどの規模で回すか*をユーザー設定が握るようになりました。

```text
# /config → Dynamic workflow size
- small  : エージェント数を絞って速く安く — 日常の反復作業
- medium : バランスの取れたデフォルト
- large  : カバレッジ最大 — 監査や徹底レビューなど網羅性が重要なとき
```

肝は**品質とコストのトレードオフがプロンプト書き換えではなく設定1行になった**ことです — 管理コンソールの Value タブ(7/5)や effort entitlements(7/3)と併せて読むと、*エージェント消費のダイヤルが個人の習慣から明示的な設定へ*と上がっていく流れの CLI 側のピースです。大量のワークフローを回す前に、自分の作業タイプごとにどのサイズが損益分岐かを一度実測しておくとよいでしょう。 [GitHub v2.1.202](https://github.com/anthropics/claude-code/releases/tag/v2.1.202)

### ワークフロー実行がテレメトリに乗る — `workflow.run_id`・`workflow.name` OTel 属性 (v2.1.202)

同じリリースで **OpenTelemetry メトリクスに `workflow.run_id` と `workflow.name` 属性**が追加されました。ワークフロー単位の実行を**既存の観測パイプライン(Datadog・Grafana など)から名前と実行 ID で追跡**できます — *どのワークフローがどれだけトークンを消費し、失敗したのはどの run か*がダッシュボードのクエリで拾えます。

要点は**エージェント自動化が大きくなるほど、可観測性がそのまま運用可能性になる**ことです — 7/3 の「実際に応答したモデルをログに残せ」とまったく同じ場所で、今回は観測対象がモデルではなく**ワークフロー層**になりました。上の Dynamic workflow size と組み合わせれば、*サイズを変えながら run ごとのコストと成果を対照する*実測ループが作れます。 [GitHub v2.1.202](https://github.com/anthropics/claude-code/releases/tag/v2.1.202)

## 開発者ワークフローティップス

### レビューコマンドを2段で運用する — 日常は `/review`、ゲートは `/code-review`

v2.1.202 で **`/review <pr>` が高速なシングルパスに回帰**しました — マルチエージェントレビューが必要なら **`/code-review <level> <pr#>`** を明示的に使います。一時期 `/review` が重いマルチエージェント経路を通っていたため、*軽い PR 確認にも時間とトークンがかかりすぎる*問題がありましたが、2つのコマンドの役割が再び分かれました。

```bash
# 日常の PR — 高速シングルパスで即フィードバック
/review 1234

# リリースゲートやセキュリティに敏感なブランチ — レベル指定のマルチエージェントレビュー
/code-review high 1234
```

原則は**レビュー強度を PR のリスクに合わせて明示的に選ぶ**ことです — すべての PR に同じ強度では、軽い変更には無駄、危険な変更には不足になります。スタック型スキル(7/4)と組み合わせれば、`/code-review /verify` のように*レビュー+実動作検証*をゲートだけに掛ける構成も可能です。チームドキュメントにレビューコマンドの規約があるなら、今回の挙動変更に合わせて更新してください。 [GitHub v2.1.202](https://github.com/anthropics/claude-code/releases/tag/v2.1.202)

### チーム導入は「境界のあるタスク3種」から — 規律ある環境が Claude Code を強くする

Claude Code をチームに導入する手順を整理した実務記事です(7/3)。結論が明快です — **「Claude Code は規律ある環境で最も強く、混沌とした環境で最も弱い」**。シニアの判断を置き換えるものではなく、*ガードレールの上の実行レイヤー*として扱えということです。

- **境界の明確なタスクを割り当てる**: オープンエンドな依頼ではなく、範囲の閉じた作業を渡します。分離ブランチで始め、シークレットや決済ロジックなど機微なファイルへのアクセスは制限します。
- **試験タスク3種から始める**: ① コードベースの説明、② バグ修正、③ テスト追加 — この3つで、自チームのコードベースにおける実力と摩擦点が素早く見えてきます。
- **繰り返し使うプロンプトパターンをチーム資産に**: うまくいったプロンプトや手順を個人のノウハウにせず、ドキュメント(CLAUDE.md・スキル)に固定します。
- **計測しながら拡大する**: サイクルタイム・バグ解決速度・レビュー負荷を導入前後で測り、感覚ではなく数字で拡大を判断します。

核心は**導入の成否はモデルではなく環境の規律が分ける**という点です — 検証ハーネス(7/4)や問題定義・受け入れ基準の成果物化(7/6)と同じ結論が、*チームオンボーディング手順*として現れた形です。上のレビュー2段運用や Dynamic workflow size をこのプレイブックの「計測」ステップに組み込めば、導入実験が一段と具体化します。 [Mean CEO](https://blog.mean.ceo/claude-code-news-july-2026/)

## セキュリティ・制限事項

### Fable 5 の使用量クレジット移行、本日(7/7)発効 — 未設定なら猶予なしで停止

長らく予告されてきた窓が今日閉じます。**7/1 の復帰以降、Pro・Max・Team(および一部 Enterprise)で週次使用上限の最大50%まで含まれていた Fable 5 が、本日で込み分を終え、別建ての使用量クレジットへ移行**します。確認済みのレートは **100万トークンあたり入力10ドル/出力50ドル**(API レートと同一)です。

- **アクション項目**: Claude Console で **usage credits を有効化し、月次支出上限(spending cap)を設定**してください — クレジットが有効化されていないと、**猶予期間なしで Fable 5 へのアクセスが停止**します。
- **Enterprise の注意点**: 標準 Enterprise シートにはそもそも込み分がなく、プレミアムシートも本日で猶予が終わり、クレジットが唯一の経路になります。
- **コスト感覚**: Sonnet 5(導入価格 2/10ドル)と比べると**入力5倍・出力5倍**です — Fable 5 は*難問のピンポイント投入*に絞り、日常作業は Sonnet 5 に置く2段構成が、料金表だけでも正当化されます。

要するに**今日から Fable 5 はサブスクリプションではなく従量課金**です — 7/2〜7/5 のブリーフィングで追跡してきた復帰ランプの終着点です。Fable 5 をパイプラインに組み込んでいるチームは、支出上限と併せて**クレジット枯渇時の fallback モデル**まで今日のうちに点検してください。 [TechTimes](https://www.techtimes.com/articles/319767/20260706/fable-5-subscription-ends-tomorrow-per-token-costs-who-gets-hit-hardest.htm) · [Digital Applied](https://www.digitalapplied.com/blog/claude-fable-5-usage-credits-july-7-pricing-guide-2026)

### 7/6 のインシデント2件 — claude.ai 障害 約47分(Claude Code OAuth を巻き込み)、Claude Tag の GitHub 操作エラー

昨日(7/6)、インシデントが2件ありました。

- **claude.ai elevated errors (19:18〜20:05 UTC、約47分)**: claude.ai が**主要障害(major outage)**の水準に達し、**Claude Code の OAuth サインイン経路も一緒に塞がりました** — Downdetector への報告は一時**2,000件**を超えました。既にログイン済みのセッションよりも、*新規サインイン・再認証*が塞がる形でした。
- **Claude Tag の GitHub 操作エラー (14:49 UTC 開始)**: Claude Tag が行う GitHub 操作呼び出しのエラー率が上がり、**Claude Code が性能低下(degraded)**と表示されました。執筆時点で**調査中(investigating)**です。

教訓は**認証経路も単一障害点**だということです — 6月末から続く短時間インシデントのパターン(6/30・7/2・7/3)が、今回は*モデルではなくプラットフォームと認証のレイヤー*で起きました。無人パイプラインでは、トークン失効が障害の時間帯と重なる場合に備えて**リトライ+失敗アラート**(7/4 で扱った v2.1.199 のリトライスタック)を維持し、Claude Tag で GitHub 自動化を回しているチームは、ステータスページで解消を確認してから再開してください。 [Claude Status](https://status.claude.com/history) · [GV Wire](https://gvwire.com/2026/07/06/claude-ai-goes-down-for-thousands-monday-downdetector-shows/)

## 知っておくと便利な小さな変更点

v2.1.202 の細かいながら実用的な修正と、リマインダーです。

- **Ctrl+R 履歴検索クラッシュを修正**: インライン履歴検索(Ctrl+R)がクラッシュする問題を解消 (v2.1.202)
- **バックグラウンドセッションの `/rename` が維持される**: 名前を変えてもジョブ再起動で巻き戻る問題を修正 (v2.1.202)
- **mTLS 証明書ローテーション時の瞬断を修正**: 証明書交換中の一時的な handshake 失敗を解消 — 企業プロキシ・mTLS 環境にはうれしい修正 (v2.1.202)
- **Remote Control の修正3種**: インタラクティブセッションでのコマンド失敗、画像・ファイルがキャプションなしで欠落、`/remote-control` の権限モード表示を修正 (v2.1.202)
- **SSH サインイン URL の整形を修正**: SSH 接続でログイン URL が壊れる問題を解消 (v2.1.202)
- **音声入力の無限リトライを修正**: キャプチャ失敗時に無限リトライループへ陥る問題を解消 (v2.1.202)
- **多数 worktree でのセッション再開が高速化**: git worktree が多いとセッション再開が遅くなる問題を修正 — 7/4 で扱った worktree 並列運用者に直接届く改善 (v2.1.202)
- **インストーラーのダウンロードを堅牢化**: 接続断でダウンロードが失敗する問題を修正 (v2.1.202)
- **本人・年齢確認は明日(7/8)発効 D-1**: 対象は個人 Free・Pro・Max のフラグ付き少数アカウントのみで、Enterprise・Team・API は除外 — 詳細は 7/5 のブリーフィング参照

## おすすめコラム＆読み物

- **「Less is More, Generally」**: **AI で何でも速く作れる時代に、量を増やすことはより良い結果につながらない**という論旨の記事です。追加は AI のおかげでほぼ無料になりましたが、**削除には依然として意図的な判断と結果への理解**が必要で、シンプルさと明瞭さは認知負荷を下げ、信頼まで生みます。ツールが強くなるほど、核心的なスキルは*何を作らないかを見極める眼*へ移るという結論 — 7/6 の「設計者としての開発者」(問題定義・受け入れ基準の所有)とまったく同じ場所を、*プロダクト感覚*の側から突いた記事です。エージェントが量産する機能案を前に、*断る基準*を持っているかを問い直させます。 [Jakub](https://jakub.kr/writing/less-is-more)
- **「Figma は次に何をするのか」**: **キャンバスは抽象にすぎず、本当のプロダクトはコード**という前提から Figma の岐路を分析した記事です。AI が*視覚的な意図と動くソフトウェアの距離*を潰していくにつれ、ボトルネックは「作れるか」から**「組織は正しいインターフェースが作られていると信頼できるか」という調整(coordination)問題**へ移った、という診断が鋭い。著者は code-to-canvas のような*現実をキャンバスへ引き戻す*方向に懐疑的で、キャンバスを**コードとランタイムという真実を覗き込むレンズ**として再定義せよと提案します — 昨日の shadcn/ui の Base UI 移行(マイグレーションをエージェントスキルとして配布)と重ねて読むと、*デザインシステムがドキュメントから実行可能な知能へ*移る流れの思考地図になります。 [Robin Cannon](https://www.robin-cannon.com/p/what-does-figma-do-next)

## 注目プロジェクト＆ツール

- **DRIFT — Mac と NVIDIA を束ねて1つの LLM を回す分散推論**: **1つの LLM をデコーダ層単位で分割し、Mac GPU(MPS)と NVIDIA GPU(CUDA)が混在する異種ハードウェアに分散**して実行するプロジェクトです。ベンダー中立の TCP+msgpack 通信、X25519・ChaCha20-Poly1305 暗号化、Ed25519 署名による作業検証、ノード離脱時の re-split・replay 復旧、OpenAI 互換 HTTP API まで備えます — Qwen は最大28台、Gemma は35台まで対応しますが、**実効的な最適域は2〜4台**で、速度より正確性を優先すると明言しています。*フロンティアモデルへのアクセスが狭まるとき、個人のリソースを束ねて能力を高める*という問題意識が — 7/5 のローカル LLM ハードウェアガイド(2千ドル vs 4万ドルのティア)に直結します: 1台を大きくする代わりに、*手持ちの機材を束ねる*という第3の答えです。 [GitHub](https://github.com/TaewoooPark/DRIFT)
- **Jetendard — JetBrains Mono Nerd Font + Pretendard の韓国語コーディングフォント**: 韓国語コーディングフォントの古くからのジレンマ — *フォールバックフォントが韓国語を描くと罫線文字の整列が崩れ、韓国語を組み込むと視覚バランスが崩れる* — を、**JetBrains Mono Nerd Font に Pretendard を合成**して解いたフォントです。韓国語グリフを拡大して不要な字間を減らすアプローチで、Yeomiel フォント(Geist Mono ベース)の手法を JetBrains Mono に移植しました。ターミナルで Claude Code を使う CJK 圏の開発者なら — トランスクリプトの罫線文字・Nerd Font アイコン・CJK コメントが1画面に混在する環境だけに — すぐ体感できる実用ツールです。 [GitHub](https://github.com/kuskhan/jetendard)
