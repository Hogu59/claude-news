## 主要な新機能と実践活用

### `/doctor` が「フルのセットアップ点検」に + agent view が状態を平易な言葉で要約 (v2.1.205)

v2.1.204/205 は修正中心のペアですが、日々の使用に直接触れる改善がいくつかあります。

- **`/doctor` フル点検**: `/doctor` が単なる診断を超え、**設定の問題を診断して修正できるフルのセットアップ点検**になり、**`/checkup` がエイリアス**として付きました。起動時の「claude command missing or broken」警告も `/doctor`・`/status` へ移りました(7/8 のブリーフィング)ので、インストール・環境チェックの単一入口として `/doctor` を使えます。
- **分類器ヘッドラインの agent view**: `claude agents` の各行が、**生のツール呼び出しテキストではなく、色分けされた状態語(working / needs input / completed)と分類器が書いた1行のヘッドライン**を表示します。セッションを開くと、**ブロック中セッションが何を尋ねているか(exact ask)まで含む全体状態**が先に出るため、複数のバックグラウンドセッションを並列で回すとき *どのセッションがなぜ止まったのか* が一目で読めます。PR を編集・マージ・コメント・プッシュするセッションも `claude agents` で該当 PR をリンクするようになりました。
- **`Claude Browser` MCP 名の予約**: 近く予定される Claude Desktop のペイン改名に先立ち、**`Claude Browser`・`Claude Preview` の MCP サーバー名が予約**され、ユーザー定義サーバーがこの名前で登録できなくなりました — デスクトップ側にブラウザ/プレビュー統合が準備中というシグナルです。

要点は **「このペアのユーザー向け改善は『運用の可視性』に集中している」** ことです — 7/8 のログイン失効警告・手動モードバッジに続き、*いま何が動き、何がブロックされているか* をより明確に見せる方向です。バックグラウンド・並列運用者なら、新しい `claude agents` のヘッドラインだけでも体感が大きいはずです。 [GitHub v2.1.205](https://github.com/anthropics/claude-code/releases/tag/v2.1.205)

## 開発者ワークフローティップス

### 無人エージェントの「完全性デフォルト」がもう一層敷かれた — 偽の承認・改ざん・rm -rf (v2.1.205)

v2.1.205 は無人実行の **完全性(integrity)** を狙った安全項目を3つデフォルトに追加しました。誰も見ていないパイプラインを回すなら、そのまま点検表になります。

- **偽の承認防止**: バックグラウンドタスク通知が **人間の入力が実際になかったことを明示**します — トランスクリプト内に *捏造された承認* が仕込まれても、それを本物の承認として実行しないよう防ぎます。
- **トランスクリプト改ざんブロック**: **auto mode ルールがセッションのトランスクリプトファイルへの改ざんをブロック**します — 監査・再現の根拠となる記録そのものをエージェントが触れないようにします。
- **`rm -rf` ガード**: auto mode が **コンテキストから値を解決できない変数への `rm -rf` 実行前に確認**を求めます — `rm -rf $UNSET/` のように空変数がルートを消し去る古典的な事故を防ぐガードです。

要点は **「インフラが静かな失敗を防いでも、エージェントの『虚偽の完了・自己承認』は別途封じる必要がある」** ことです — 7/4 のサブエージェントの静かな失敗修正、「検証ハーネスに投資せよ」と同じ場所で、今回は *記録の信頼性* をリリースが支えます。無人自動化に自前の承認・監査ロジックを載せているなら、この3つのデフォルトと二重にならないか一度点検してください。 [GitHub v2.1.205](https://github.com/anthropics/claude-code/releases/tag/v2.1.205)

### サブエージェントは「3〜5個」がスイートスポット — CLAUDE.md は長さがそのまま性能予算

サブエージェントをうまく使う実務知見を、データとともに整理したベストプラクティスガイドが今週いくつも出回りました。7/8 の *リサーチ系サブエージェントは Haiku で* と重ならない、規模とコストの結論だけ抜き出すとこうです。

- **並列サブエージェントは3〜5個がスイートスポット**: 大半のタスクで3〜5個が最適で、**10個を超えると追加の利得はまれで、コストだけが大きく増します**。各サブエージェントは別々のコンテキストウィンドウを持つため、**サブエージェント多用のセッションは単一スレッドの約7倍のトークン**を消費し得ます。
- **CLAUDE.md は「多く」ではなく「短く」**: CLAUDE.md が長くなるほど **指示が無視される確率が上がります** — 簡潔さは好みではなく性能要件です。*必ず毎回起きるべきルール* は文章で持たず **フックへ移して**強制し、CLAUDE.md には判断が要る規約だけ残しましょう。

教訓は **「委譲とメモリはタダではない」** ということです — サブエージェントの並列性も、CLAUDE.md の文章も、それぞれトークンと注意の予算を使います。上の v2.1.205 の完全性デフォルトが *委譲の安全* を支えるなら、この習慣は *委譲の経済性* を守ります。 [Claude Code ベストプラクティス](https://code.claude.com/docs/en/best-practices)

## セキュリティ・制限事項

### 中国 NVDB が Claude Code の「バックドアのリスク」を警告、Anthropic は「3月の実験、次リリースでロールバック」と応答 (7/8)

**7/8、中国工業情報化部(MIIT)傘下の国家情報セキュリティ脆弱性データベース(NVDB)が、Claude Code には重大な脅威となるバックドアのリスクがあると警告**し、機関に即時点検とアップグレードまたは削除の検討を勧告しました。3者の主張をそのまま並べると次の通りです。

- **NVDB の主張**: この機能が **ユーザーの位置や身元関連の識別子など機微情報を、同意なく Anthropic のサーバーへ送信**し得るというものです。当局は即時の全面点検を勧告しました。
- **Anthropic 側の応答**: Claude Code エンジニアの **Thariq Shihipar** は X で当該トラッキング機能を認め、*これは3月に始めた実験で、認可されていないリセラーによるアカウント不正利用を防ぎ、distillation を防御するためのもの* だと述べました。すでにより強力な解決策を導入済みで、**次のリリースでこの機能を完全にロールバックする計画**だと付け加えています(つまり本ブリーフィング時点でロールバックは予告であり、完了ではありません)。
- **Alibaba の対応**: 中国の Alibaba はセキュリティ上の懸念を理由に、**7/10 から社内での Claude Code 利用を禁止**すると従業員に通知しました。Anthropic の利用ポリシーはもともと、*中国に本社を置く組織が過半を所有する主体* による利用を禁じています。

**影響を受けるバージョン範囲は報道により 2.1.91〜2.1.196(4月〜6月)と伝えられています**が、NVDB の元の勧告自体はこの場で検証できておらず、一部の報道(CBS)は特定のバージョンを明示していません — さらにエンジニアの言う *3月の実験* と報じられた *4月2日の開始* に時期の食い違いがあるため、バージョン範囲は断定せず、**元の勧告と次リリースのロールバック有無を直接確認**するのが安全です。本件は6〜7月を通じて追ってきた **distillation 防御の文脈**(Fable 5 セーフガード・CJS フレームワーク)と同じ流れに位置しますが、*同意なき位置・身元テレメトリが不正利用防止として正当化されるか* は政府と企業で見解が分かれる点なので、どちらの肩も持たず、上記3つの事実で各自が比較衡量するのが妥当です。 [CBS News](https://www.cbsnews.com/news/china-security-backdoor-anthropic-ai-coding-tool/) · [CNBC](https://www.cnbc.com/2026/07/08/china-anthropic-ai-claude-code-backdoor-security-threat.html)

### GitLost — GitHub の「AI エージェント」を騙して非公開リポジトリを流出 (Claude Code ではない)

Noma Security が公開した **GitLost** は、**Claude Code ではなく GitHub の AI エージェント**を狙った間接プロンプトインジェクションの事例です — 誤読しないよう明記します。着眼点は *エージェントが信頼された指示とユーザー制御コンテンツを同じコンテキストウィンドウで扱う* という構造的弱点です。

- **手口**: 公開リポジトリに **もっともらしい issue** を仕込み、*Additionally,* のようなキーワードでガードレールを回避します。issue の割り当て(assign)イベントでトリガーされた AI エージェントに、**非公開リポジトリの内容(例: README)を取得させ、公開コメントとして投稿**させ、誰でも読める状態に露出させます。
- **根本原因**: エージェントに **公開・非公開リポジトリへのアクセスを同時に与えつつ、結果を公開できる**ようにした設計です — *ユーザー制御コンテンツを信頼された指示入力として扱ってはならない* という原則の違反です。

教訓は **「ツール権限と出力チャネルを分離せよ」** という古くからのものです — 7/8 の SkillCloak(静的スキャナ回避)とともに、*コーディングエージェント全般* を貫くプロンプトインジェクション脅威の今週2件目です。Claude Code であれ GitHub のエージェントであれ、**読めるものと外へ送れるものの境界**を明示的に狭めておく習慣が防御の出発点です。 [Noma Security](https://noma.security/blog/gitlost-how-we-tricked-githubs-ai-agent-into-leaking-private-repos/)

## エコシステム＆プラグイン

### TypeScript 7.0 — tsc が Go ネイティブへ、フルビルド8〜12倍 (Microsoft)

**Microsoft が TypeScript 7.0 を発表**しました — 中核は **ツールチェーンを TypeScript 自身の実装から Go ネイティブコードへ移植**したことです。結果は劇的で、**フルビルドで8〜12倍の高速化**(VS Code ビルドが125.7秒→10.6秒、11.9倍)、メモリもプロジェクトごとに6〜26%削減。新しい `tsc` は **LSP ベースの言語サーバー**で VS Code・Visual Studio・WebStorm のエディタ支援を統一し、`--checkers`・`--builders` フラグで **並列型チェック**を有効にします。ただし **安定したプログラミング API はまだなく(7.1 で提供予定)**、typescript-eslint・Vue・Svelte などは当面 TypeScript 6.0 を併存させる必要があります。

Claude Code 視点の含意は直接的です — エージェントが TS リポジトリで絶えず回す **型チェック・コンパイルのループが丸ごと速くなり**、Claude Code の LSP ツールが依拠する言語サーバーもこのネイティブ実装の上に載ります。7/6 で扱った **@ttsc/graph**(TS コンパイラのパース済み構造を再利用してトークンを節約する MCP)と併せて読めば、*TS コンパイラ基盤が速くなるほど、その上のエージェントワークフローも速くなる* という流れの根幹をなす部品です。大型 TS モノレポで Claude Code を回すチームなら、検証ゲート(型チェック)の遅延がボトルネックだった箇所ほど大きな変化です。 [TypeScript 7.0 発表](https://devblogs.microsoft.com/typescript/announcing-typescript-7-0/)

## 知っておくと便利な小さな変更点

v2.1.204/205 の細かいながら実用的な修正とリマインダーです。

- **Windows の worktree 削除によるデータ損失を修正**: worktree 内に **NTFS ジャンクション・ディレクトリシンボリックリンク** があると、worktree 削除が **worktree 外のファイルまで消していた**深刻な問題を解消 (v2.1.205)
- **`--json-schema` 無効スキーマの静かな失敗を修正**: スキーマが無効なとき **非構造化出力を黙って出していた**問題、`format` キーワードを使うスキーマが拒否されていた問題を解消 (v2.1.205)
- **`--max-turns` 上限でのメッセージ喪失を修正**: Claude が作業中に送ったメッセージが **上限でターンが終わる際に静かに消えていた**問題を解消 (v2.1.205)
- **自動アップデートのメモリを約400MB削減**: アップデートバイナリのダウンロードをメモリバッファリングではなく **ディスクへストリーミング**し、アップデータのピークメモリを大きく低減 (v2.1.205)
- **`SendMessage` 再開後の状態表示を修正**: `SendMessage` で再開したバックグラウンドエージェントが一覧で「failed」「completed」のまま残る問題、読めるテキストがないターンで「needs input」と「working」を行き来する問題を解消 (v2.1.205)
- **`claude mcp add-from-claude-desktop` の堅牢化**: サーバー名に非対応文字があると止まる問題を修正し、**無効な名前だけ報告して残りのサーバーはインポートを継続**します (v2.1.205)
- **Cowork VM モードのログイン修正**: CLI 2.1.203+ で Cowork VM モードのローカルエージェントセッションが「Not logged in」で起動失敗する問題を解消 (v2.1.205)
- **リマインダー**: 7/7 発効の Fable 5 使用量クレジット移行、7/8 発効のフラグ付き個人アカウントの身元・年齢確認は、いずれも現在適用中です — 詳細はそれぞれ 7/7・7/5 のブリーフィング参照

## おすすめコラム＆読み物

- **「エージェントループの始め方」 — Claude エージェントループの4類型**: Anthropic の ClaudeDevs がまとめた **エージェントループ設計パターン** の案内です。各ステップを個別プロンプトで指示する代わりに、停止条件までタスクサイクルを繰り返す *ループ* を **4類型**に分けます — **turn-based**(ユーザートリガー・手動検証)、**goal-based**(目標達成で自動停止)、**time-based**(スケジュール・間隔)、**proactive**(イベント駆動・最小介入)。中心の助言は *最も単純な解法をまず* で、複雑なパターンは選択的にのみ重ねること — ループ出力の品質は結局 **周辺システム(コード品質・トークン管理)** に依ると釘を刺します。7/5 の「ループエンジニアリングの美学」が *なぜループが重要か* を4層で描いたなら、この記事は *どのループから選ぶか* を公式ドキュメントの視点で絞った対になります。 [ClaudeDevs](https://x.com/ClaudeDevs/status/2074208949205881033)
- **「ClickHouse が観測性の戦いで勝つ理由」**: **ClickHouse が Elasticsearch・LGTM・Datadog を抑え、観測性インフラの勝者になりつつある理由**を実務視点で分析した記事です(Mat Duggan)。論旨は — ログは結局 *大規模・非定型・予測不能なクエリ* に収束し、ClickHouse の **列指向ストレージと10〜14倍の圧縮効率**がこのロギングパターンと自然に噛み合い、**1TB/日から10TB/日以上へデータが増えても運用のシンプルさが一貫して保たれる**というものです。7/7 の workflow OTel 属性(観測可能性がそのまま運用可能性)と併せて読めば、*エージェント自動化が吐くログ・トレースをどこに蓄えるか* という次の問いに触れます。観測パイプラインを再設計中のチームなら、ストレージ層選定の根拠として読む価値があります。 [Mat Duggan](https://matduggan.com/clickhouse-is-winning-the-observability-wars/)

## 注目プロジェクト＆ツール

- **Ternlight — ブラウザ(WASM)で動く5〜7MB の埋め込みモデル**: **テキスト埋め込みと類似度マッチングをブラウザ内で WebAssembly により完全に実行**するセマンティック検索ツールです。テキストを **384次元ベクトル**に変換し、サーバー呼び出しなしで意味的に近い文書を見つけます。驚くのは圧縮率で、**推論エンジン・モデル・BERT トークナイザーを合わせて5〜7MB(gzip)**に収めました。**三値量子化**(重みを -1・0・+1 に制限)と **Rust ベースのカスタム SIMD エンジン**で CPU のみで動作し、ブラウザ・エッジランタイム・オフラインアプリ・IoT までカバーします(`@ternlight/base` 7MB・埋め込みあたり約5ms、`@ternlight/mini` 5MB・約2.5ms)。Node 18+・ブラウザ・Cloudflare Workers・Vercel Edge・Deno・Bun をサポート — *プライバシー・オフライン・エッジ検索* を **30倍圧縮**で解いた事例で、ローカルファーストなツールを組み込む開発者にすぐ実用的です。 [GitHub](https://github.com/soycaporal/ternlight)
