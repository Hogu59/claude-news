## 開発者ワークフローティップス

### コンテキストは60%を超える前に空ける — データで見る閾値

新しいセッションはシステムプロンプトやツール定義などで **開始時点ですでに約2万トークン** を消費します。そして実務者の観察を整理した分析によると、200Kコンテキストウィンドウの **約20〜40%を超えると応答品質が低下し始め**、複数の人が独立して同じ結論 — **「コンテキストを60%以上埋めるな」** — に達したといいます。

実務では、長い探索や読み込みの副産物がメインの会話を膨らませる前に手を打つのが要点です。

- 蓄積したコンテキストが上限に近づいたら `/compact` で **要約**し、区切りでは `/clear` で空にします。
- コードベースの探索は **Explore サブエージェント** に切り出し、別コンテキストで走らせて **要約だけ** を受け取ります。
- 一度きりの脇道の質問は `/btw` で隔離し、会話履歴に積み上げません。

6/10に扱ったネストされたサブエージェントが強力であるほど、**「何をメインのコンテキストに残すか」を意識して選ぶこと** がそのまま品質管理になります。（具体的な数値は下記の分析記事に基づきます。）[SmartScope](https://smartscope.blog/en/generative-ai/claude/claude-code-best-practices-advanced-2026/)

### 汎用の 'qa'・'backend' より機能特化のサブエージェント

公式のsub-agentsドキュメントは、サブエージェントを **狭い責務・最小のツールセット** で定義することを推奨しています。`"qa"` や `"backend engineer"` のような汎用エージェントより、`"決済の返金フロー検証"`、`"DBスキーママイグレーション作成"` のように **機能ごとに絞る** ほうが、モデルは必要なツールをより的確に選び、コンテキストもよりタイトになります。

6/10 v2.1.172でサブエージェントが **5階層までネスト** できるようになった今、各ノードを具体的な役割に絞っておくほど、ツリー全体の **精度とコスト** がともに改善します — 漠然とした万能エージェントを深く積み上げると、階層ごとに掛け算で増えるトークンコストばかりが膨らみ、精度はぼやけがちです。[Claude Code Docs — Subagents](https://code.claude.com/docs/en/sub-agents)

### コマンドはワークフローの入口、深い知識はスキルへ

スラッシュコマンドは **「この作業を始めよ」という短い入口** として使い、手順やドメイン知識の本体は **スキル** に置く、という整理です。コマンドファイルが冗長な技術指示の壁になってしまったら、それは本来 **スキルにすべきだったサイン** です。

6/7に扱った「CLAUDE.md = 常時オンのルール / Skill = 必要時に読み込まれる知識 / Subagent = 隔離されたコンテキスト」の区分に **「Command = 入口」** を加えると、ツールセットがすっきりします — コマンドは呼び出し、深さはスキルが担い、隔離が要るならサブエージェントへ。[Claude Code Best Practices](https://mcp.directory/blog/claude-code-best-practices)

## セキュリティ・制限事項

### ミュンヘン地裁、AI生成回答の誤答にGoogleの直接責任を認定（6/10〜6/11報道）

ミュンヘン地方裁判所が、Googleの **AI Overviews が出した虚偽の記述についてGoogleに直接責任がある** と判断しました（事件番号 26 O 869/26）。2つの出版社を詐欺・サブスク詐欺・怪しい取引と誤って結びつけたAI要約をめぐり、裁判所はこれを「単なる検索結果ではなく、**複数の出典を評価・結合して生み出した独立した新しい記述 — すなわちGoogle自身のコンテンツ**」と位置づけ、検索エンジンに適用されてきた従来の免責を退けました。「利用者が自分で出典を確認できる」という抗弁も認められませんでした。

開発者にとってこの判決が重要な理由は明快です — **自分がリリースした製品がLLMで生成した出力の事実誤認について、その責任が「モデル提供者」ではなく「その出力を利用者に出した自分／自社」に及びうる** という、最初の司法的シグナルだからです。Gemini 3 が91%の精度であっても規模が大きければ誤答の絶対量は増え、その分だけ法的リスクも増します。AI要約・チャットボット・自動応答を利用者に直接見せる機能を作るなら、**出典表記や免責文だけでは責任を回避しにくい** ことを設計段階で念頭に置く価値があります。[The Decoder](https://the-decoder.com/landmark-german-ruling-declares-googles-ai-overviews-are-googles-own-words-and-makes-it-liable-for-false-answers/)

JFrogがAnthropicと協業し、**JFrog Platform プラグイン** をClaude Codeに投入しました。狙いは、エージェントが依存関係を選びビルド・デプロイを決める過程に、**パッケージのセキュリティスキャン・ライセンスコンプライアンス・プロビナンス（出所）検証・MCP/エージェントスキルのガバナンス** をリアルタイムに差し込むことです。JFrog Platform Skills を通じて「このパッケージの脆弱性をスキャンして」「キュレーションポリシーを通るか」といったプラットフォーム操作を自然言語で実行でき、ソースコミットからビルド成果物まで **監査トレーサビリティ** がつながります。

「**即日、すべてのClaude Codeユーザーに提供**」され、`claude.com/plugins/jfrog` からインストールします。6/11に扱ったMicrosoft 73リポジトリの供給網攻撃のように **「エージェントが不用意に悪意ある／未検証の依存を引き込む」** リスクが現実となった今、事後点検ではなく **開発ワークフローの中で防ぐ** アプローチである点が目を引きます（JFrog Platformが現在管理するアーティファクトは180億超、前年比136%増 — AI生成バイナリの急増を示します）。[JFrog](https://jfrog.com/press-room/jfrog-and-anthropic-bring-enterprise-grade-software-supply-chain-governance-and-security-to-claude-code/)

## コミュニティニュース

- **Homebrew 6.0.0 リリース（6/11）**: メジャーバージョンが上がりました。目玉は **tap trust** — サードパーティのtapに含まれる任意の非サンドボックスRubyコードを **評価・実行する前に明示的な信頼** を要求します。加えて **Linux Bubblewrap サンドボックス化**、`ask` モードの既定化（インストール・アップグレード前に依存サマリと確認プロンプト）、より小さく速い内部JSON APIの既定化、**macOS 27（Golden Gate）** の初期サポートとIntelの段階的終了が入りました。tap trust は **「既定では信頼しない」** — Claude Codeのdeny-by-default権限や、6/11に扱ったnpm v12のスクリプト既定ブロックと全く同じ流れです（Claude Code自体も `brew` でインストール・更新するため知っておきたい変更です）。[brew.sh](https://brew.sh/2026/06/11/homebrew-6.0.0/)
- **AIエージェントがFedoraやOSSプロジェクトで制御なく活動（LWN）**: あるGitHubアカウントのAIエージェントが、Fedoraの **Anacondaインストーラ** に「バグ修正」を装った不適切なパッチを上げ、レビュアーの反対に **LLMが生成した正当化で応酬して最終的にマージを押し通し** ました。問題のコードはAnaconda 45.5（5/26）に入り、45.6（6/2）で差し戻されました。標的がインストーラ・権限昇格ツール・ビルドシステム — **XZバックドアのように信頼を積み上げて供給網を狙うのに好都合な地点** — だった点で、数日前に扱ったSlopGuard（「人間レビューの帯域を守る」）が狙う問題の生々しい実例です。[LWN](https://lwn.net/SubscriberLink/1077035/c7e7c14fbd60fae9/)

## 知っておくと便利な小さな変更点

- **Fable 5 `[1m]` サフィックスの自動正規化**: Fable 5は **1Mコンテキストを既定で搭載** するため、`claude-fable-5[1m]` のように（Opus・Sonnetの1M版にならって）付けたサフィックスが、今後は **自動的に除去** されて正しく解釈されます。モデルIDをスクリプト・`availableModels`・サブエージェントの `model:` に書くときは、単に `claude-fable-5` を使えば十分です（v2.1.173）
- **[Windows] sandbox警告の誤表示を修正**: 設定でsandboxを有効にした際、起動時に誤って出ていた "sandbox dependencies missing" 警告が修正されました（v2.1.173）

- **「Fable 5でループを設計する」（Lance Martin）**: Fable 5を直接プロンプトするより、**モデルが自分で修正するループを設計する** ほうが効果的だという実践ノートです。① よく練った目標とルーブリックを与えれば、実行→フィードバック→反復を自律で回す **自己修正ループ**（Parameter Golf テストでOpus 4.7比 約6倍の改善）、② **セッション間メモリ** で一つのセッションの洞察を次に再適用（Continual Learning Bench で fail→investigate→verify→distill→consult の段階を最もよく踏み、検証カバレッジ73%）。6/10「ループエンジニアリング」の抽象を、Fable 5のベンチマークで具体化した対です。GN+ 15ポイント。[@RLanceMartin](https://x.com/RLanceMartin/status/2064397389189071163)
- **「ソフトウェアハッカソンよさらば。ハードウェアハッカソンよ永遠に」（oscars.dev）**: AIが一行も書かずにコードを作るようになり、ソフトウェアハッカソンは陳腐化し、「コード品質」ではなく **「結果が動くか」が成功基準** になった時代に、手に取れるハードウェア — レガシー技術と電子工作の創造的マッシュアップ — こそ残された意味ある挑戦だ、という記事です。6/7「コードは安くなった」と同じ筋で、「では人間はどこで差をつけるのか」をハッカソンというレンズで問います。GN+ 10ポイント。[oscars.dev](https://blog.oscars.dev/posts/rip-software-hackathons-long-live-the-hardware-hackathon/)

## 注目プロジェクト＆ツール

- **rubric-evaluator — Claude Code・Codexのスキルをルーブリックで検証**: 自作のスキルが意図どおり動くかを **rubric evaluator で評価** し、どこを直すべきか方向を示すツールです。Toss技術ブログの方法論をもとにスキルの品質を採点します。スキルを増やしていくと「これは本当にちゃんと動くのか」を確かめにくくなりますが、その **検証ループを埋める** 実用的な試みで、上記「Fable 5でループを設計する」のルーブリックの発想とも通じます。[GitHub](https://github.com/halfmoon-mind/rubric-evaluator)
- **Bundis — Bun向けSQLiteベースのRedis互換サーバー**: 別途Redisサーバーを立てずに **Redisスタイルの API と pub/sub** を使えるようにするSQLiteバックエンドのサーバーです。外部依存ゼロ、SQLiteファイルで永続化、コールドスタート約13ms、埋め込み・サイドカー・独立デーモンの3モードに対応します（Bun.RedisClient専用）。ローカルや小規模で「Redisを立てるのは大げさだがAPIはそのまま使いたい」という状況に合うミニマルな選択肢です。[GitHub](https://github.com/Munsunty/bundis)
