## 主要な新機能と実践活用

### Claude Design ↔ Claude Code が双方向連携 — `/design` で仕様を作り、そのまま実装へ渡す（6/17）

Anthropic がデザインアシスタント **Claude Design** の大型アップデートを 6/17 に出荷し、**Claude Code との連携を双方向に**しました。6/17 に扱った Head of Design **Meaghan Choi** の「作業の仕方」（複数の案を作って選び正当化する）が一個人の*習慣*だったのに対し、これはプロダクトが実際に**コード↔デザインの橋を両方向に**出荷した変化です。

- **Claude Design → Claude Code への受け渡し**：Claude Design が案を仕上げたら、デザインを**そのまま Claude Code に渡して**実装させます — Anthropic の表現で「最初から作り直すことなく」インターフェースをコーディングします。
- **ローカルのコードベースで作業**：Claude Design が**既存のコードベース上で直接**作業するようになり、生成されるアセットが**実際のプロダクションコードと食い違わない**ように揃えます。
- **逆方向 — Claude Code 内で `/design`**：Claude Design を別途開く必要はなく、**Claude Code で `/design` と入力**すれば、コーディングのインターフェース内でそのままデザインを作れます。
- **デザインシステムの import**：**GitHub・raw ファイル**からデザインシステム全体を取り込み、新規作業の土台にします。

```bash
# Claude Code のセッション内でそのまま案を作る
/design 決済完了画面 — 既存のデザイントークンに従い、レシートカードを追加
```

実務での効果は明快です。これまで「画面を1つ作って」と投げると、エージェントが UI を**即興で組み立て**、レイアウト・間隔・状態をその場で推測して、視覚的な一貫性が崩れがちでした。今は仕様 → 実装が**コードベースのデザインシステムを基準に**一筋でつながり、**ターミナルを離れずにデザインのループを閉じられる**のが核心です。Claude Design は **claude.ai/design** またはデスクトップアプリのサイドバーからアクセスでき、Pro・Max・Team・Enterprise 向けのベータです。[Engadget](https://www.engadget.com/2196329/anthropics-design-assistant-now-works-better-with-its-coding-agent/) · [TechRepublic](https://www.techrepublic.com/article/news-anthropic-claude-design-overhaul-enterprise-teams/)

## 開発者ワークフローティップス

### デザインを先に確定して「渡す」— エージェントに UI を即興させない

上の連携を実務の習慣に落とすには、順序が大事です。「この画面を作って」をいきなり投げると、エージェントはレイアウト・間隔・状態をその場で**推測**し、結果をレビューする段階で視覚的な決定が毎回ぶれます。代わりに**視覚仕様を先に固めてから**実装へ渡しましょう — Claude Design で（または Claude Code の `/design` で）案を作り、**その1つを根拠とともに選び**、確定した版を Claude Code の実装に受け渡します。

核心は 6/17 の Meaghan Choi の原則と同じです — **「誰でもビルドできるからといって、すべてを出荷すべきとは限らない」**、LLM は依然としてデザイン判断が苦手だ、ということ。実行（コーディング）は委譲しつつ、**何を作るか（視覚の決定）は人が先にロック**しておけば、実装段階でエージェントが推測で枝分かれするコストが消えます。漠然と「UI を作って」ではなく、**仕様を先に確定 → 実装を委譲**の二拍子で区切ってください。[Engadget](https://www.engadget.com/2196329/anthropics-design-assistant-now-works-better-with-its-coding-agent/)

### サブエージェントに「最小権限」を与える — 役割ごとに tools を絞って git にコミット

6/18 に扱った「サブエージェントは1つにつき1役割」が*description/役割*の分離だったとすれば、その対になる実務は **`tools:` フィールドの分離**です — 各サブエージェントに**その仕事に必要な道具だけ**を与える最小権限（least-privilege）の設計です。コードレビューのエージェントはファイルを読むだけなので Bash・Write は不要ですし、ドキュメントのエージェントはシェルを回す用がありません。

```markdown
---
name: code-reviewer
description: 変更されたファイルのバグ・規約違反だけをレビュー
tools: Read, Grep        # 読み取り専用 — Bash・Write なし
model: claude-sonnet-4-6
---
```

`tools:` を絞ると2つを同時に得られます — **集中**（余計な道具がないので的外れな行動が減る）と**安全**（レビューのエージェントは*物理的に*シェルを回したりファイルを変更したりできない）。この定義を **project-scope（`.claude/agents/*.md`）に置いて git にコミット**すれば、チーム全体が同じガードをそのまま継承します。6/14 の「サブエージェントの `model:` を repo にコミット」と同じ場所 — 役割・モデル・権限をすべてコードとして埋め込むパターンです。[PubNub — Claude Code サブエージェントのベストプラクティス](https://www.pubnub.com/blog/best-practices-for-claude-code-sub-agents/)

## セキュリティ・制限事項

### Mythos 輸出規制の「本当の引き金」は SK テレコムだった — 優先アクセス名簿と中国連携の「疑い」（Wired、6/18）

6/12 の **Fable 5・Mythos 5 回収**の発端が、1社に絞られました。**Wired の報道**によれば、ホワイトハウスが Anthropic に回収を命じた直接の対象は、韓国最大の通信会社 **SK テレコム**でした — 数日来追ってきた 6/18 の「中国系グループが Mythos にアクセスした疑い」という筋が、この報道で**具体的な1企業の名前**として現れた形です（矛盾ではなく、その筋の解消）。

順序が肝心です。

1. **6/4**：SK テレコムが Anthropic の **Project Glasswing 参加**とともに、**Claude Mythos への早期アクセス**を確保したと発表しました（SKT は Anthropic の投資家でもあります）。
2. **Amazon が Fable 5 の jailbreak を報告する「数日前」**、ホワイトハウスは Anthropic が提出した**優先アクセス（priority access）名簿**から **SK テレコム1社**を名指ししました — **中国との関係が疑われる**という理由で、政府は Anthropic が技術を守り切る能力への信頼を失ったとして、**SKT の Mythos クレデンシャルを即時取り消す**よう要求しました。
3. その**後で** Amazon が Fable 5 のガードレール回避を発見し、ホワイトハウスはこれを加えて、**全外国籍の Fable 5・Mythos 5 アクセスを一括停止**する輸出規制指示（6/12）に拡大しました。

**SK テレコムは疑惑を否定**しています — 中国とは何の関係もない、という立場です（SKT には2000年代の China Unicom 合弁など過去の事業実績はありますが、現在の連携はないとしています）。Anthropic はワシントンに**シニアの技術人材を派遣してホワイトハウスと直接交渉**中で、双方とも**早期解決の意思**を示しており、ある幹部はアクセスが「**数日内**」に復旧しうると自信を見せました。

開発者にとっての要点は2つです。① 今回の回収が「抽象的な安全性」ではなく、**名簿上の1企業への地政学的な疑い**から始まったことが明確になりました — 規制の引き金が極めて外科的だったということです。② それでも韓国・日本などの**外国籍開発者は今も両モデルにアクセスできない**状態なので、`fallbackModel`・`availableModels` で Opus 4.8 などの代替経路を用意しておく実務対応は変わりません。[Wired](https://www.wired.com/story/sk-telecom-anthropic-mythos-export-controls/) · [Korea JoongAng Daily](https://www.koreajoongangdaily.com/business/anthropic-confident-of-reenabling-mythos-fable-5-access-in-coming-days-executive/12727522)

## エコシステム＆プラグイン

### Google の Open Knowledge Format（OKF）v0.1 — エージェントが共有するベンダー中立な知識標準（新規）

Google が AI エージェント向けの**ベンダー中立な知識標準 OKF v0.1** を新たに公開しました（今週浮上）。ある生産者が作った知識を、**翻訳レイヤーや独自統合なしに**別のエージェントがそのまま消費できるようにする、というオープン仕様です。

形式は馴染みのあるものです — 知識を **YAML frontmatter 付きの markdown ファイル**で表現し、概念1つ（テーブル・データセット・メトリクス・ランブック・API）が**メタデータを持つ単一の markdown ファイル**となって、標準の markdown リンクで互いに結ばれ**知識グラフ**を成します（Claude Code の CLAUDE.md やスキル定義と同じ「人も読める」形）。狙う問題は明快で、組織の知識がメタデータカタログ・wiki・コードコメント・担当者の頭の中に散らばり、**あらゆるエージェントビルダーが同じ「コンテキスト組み立て」問題を繰り返す**こと。OKF は「**SDK なしで生産し、統合なしで消費**」を目標に、異なるエージェント・データシステム・可視化ツールが同じ知識フォーマットの上で**ロックインなしに**一緒に動けるようにします。下の「注目プロジェクト」の **AKB** が、まさにこの OKF の上に建てられた事例です。[Google Cloud Blog](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing)

## 注目プロジェクト＆ツール

- **GitDB — GitHub リポジトリを「サーバーレス RDB」として使うライブラリ**：GitHub リポジトリを**サーバーレスなリレーショナル DB のバックエンド**として使う TypeScript ライブラリです。データは manifest・mutation log・snapshot ファイルとして **Git コミットに永続化**され、リポジトリの**履歴がそのまま監査証跡（audit trail）**になります。insert・upsert・select・deleteWhere といったテーブル操作と、**JOIN・GROUP BY を含む SQL スタイルのクエリ**、インデックスベースの取得、トランザクション、平文・暗号化ストレージに対応します。**別途 DB サーバーなしで**保存が必要なブラウザ拡張・静的アプリ・小ツールを狙っており、作者も GitHub API のレイテンシ・レート制限のため、**低頻度のアプリ・デモ・内部ツール**（リアルタイム性能よりも変更履歴とデプロイの簡潔さが重要な場面）に向くと明言しています。[GitHub](https://github.com/3x-haust/gitdb)
- **AKB（Agent Knowledge Base）— 人と AI エージェントが共に使う共有知識ベース**：人と AI エージェントが**共有ドキュメントを一緒に読み書きする**よう設計された協働知識インフラです。ドキュメントは **Google の Open Knowledge Format と互換の markdown**（上のエコシステム参照）で保存し、そこに**クエリ可能なテーブル・ファイル保存・知識グラフ**を加えています。際立つのは権限モデルで、エージェントは**特定ユーザーに紐づくトークン**で認証し、そのユーザーの vault アクセス権を**そのまま継承**、この権限は**アプリケーションと DB の両層**で強制されます。開発チームはこの上に「reef」というイシュートラッカーを載せ、開発者とプロダクトマネージャーが**技術的背景が違っても同じドキュメントで協働**するのに使っています — 上の「サブエージェントの最小権限」ティップを*組織の知識*の次元に拡張した形です。[GitHub](https://github.com/dnotitia/akb)
