## 主要な新機能と実践活用

### メインモデルが落ちても止まらない — `fallbackModel` / `--fallback-model`（v2.1.166）

v2.1.166 に、**フォールバックモデルを最大3つまで順番に指定**できる `fallbackModel` 設定が追加されました。メインモデルが過負荷（overloaded）または利用不可のとき、指定した順にフォールバックします。これまでヘッドレス専用だった `--fallback-model` フラグが、**インタラクティブセッションにも適用**されるようになりました。さらに、API が（従来はリトライ不可と扱われていた）予期しないエラーを返した場合に、**フォールバックモデルでターンを1回自動リトライ**します — ただし認証・レート制限・リクエストサイズ・トランスポートのエラーは従来どおり即座に表面化します。

```bash
# メインモデルが過負荷なら Sonnet にフォールバック（インタラクティブセッションにも適用）
claude --fallback-model claude-sonnet-4-6
```

設定ファイルでは `fallbackModel` キーで最大3モデルを順番に指定できます（正確なスキーマは、今後更新される公式 settings ドキュメントに従ってください）。6/2〜6/5 に複数モデルのエラーインシデントが続いたことを踏まえると、モデルの可用性に依存する無人パイプラインや CI で**「1つのモデルが揺らいでもジョブが死なない」**ための可用性ガードとして有用です。

[GitHub v2.1.166](https://github.com/anthropics/claude-code/releases/tag/v2.1.166)

### deny ルールに glob — `"*"` 一行で全ツールを拒否（v2.1.166）

v2.1.166 から、権限 **deny ルールのツール名の位置に glob パターン**を使えます。`"*"` は**すべてのツールを拒否**します。逆に allow ルールでは非 MCP の glob を拒否し（過剰な許可を防止）、deny ルールに未知のツール名があると起動時に警告します。

```json
// .claude/settings.json — まず全拒否し、必要なものだけ許可する "deny-by-default" の骨組み
{
  "permissions": {
    "deny": ["*"],
    "allow": ["Read", "Grep", "Glob"]
  }
}
```

厳格なサンドボックスや読み取り専用の探索エージェントを作るとき、ツールを1つずつ拒否する代わりに、**`"*"` ですべてロックし、必要なものだけ開ける**ホワイトリスト方式を一行で表現できます。（正確な優先順位や組み合わせのルールは公式 permissions ドキュメントを確認してください。）

[GitHub v2.1.166](https://github.com/anthropics/claude-code/releases/tag/v2.1.166)

## 開発者ワークフローティップス

### Anthropic が自ら公開した「スキルの上手な使い方」 — 9つのカテゴリと Gotchas セクション

Anthropic の Claude Code チームが、社内で**数百のスキルを実際に運用して得た教訓**を公式ブログで公開しました（Thariq Shihipar、6/3）。要点は次のとおりです。

- **スキルは「マークダウンファイル」ではなくフォルダ**です — スクリプト・アセット・データを同梱できます。
- 社内スキルを分類したところ、**9つのカテゴリ**に収束しました。ライブラリ/API リファレンス、製品検証、データ取得/分析、ビジネスプロセス/チーム自動化、コードのスキャフォールディング/テンプレート、コード品質/レビュー、CI/CD・デプロイ、ランブック（runbook）、インフラ運用です。
- **1つのスキルは1つのカテゴリに。** 良いスキルは1つのカテゴリにきれいに収まり、欲張ったものは複数にまたがってエージェントを混乱させます。
- **最も価値ある内容は「Gotchas」セクション**です。Claude がそのスキルを使うときに繰り返しはまる落とし穴を集め、エッジケースを見つけるたびに更新しましょう。
- **知識系スキルは当たり前のことを書かない。** モデルがすでに知っていることを繰り返さず、**Claude を普段の思考パターンの外へ押し出す**情報に集中します。
- **段階的開示（progressive disclosure）。** すべてを前もって詰め込まず、状況別の参照ファイルを指し示して**必要なときに読ませ**ます。

マーケットプレイスも中央委員会が決めるのではなく、GitHub のサンドボックスフォルダに上げ、**有用性が実証されたら PR で昇格**する有機的な方式だそうです。

[Anthropic — Lessons from building Claude Code: How we use skills](https://claude.com/blog/lessons-from-building-claude-code-how-we-use-skills)

### 大量・機械的な作業では thinking をオフにしてコストとレイテンシを削減 — `MAX_THINKING_TOKENS=0`（v2.1.166）

v2.1.166 から、`MAX_THINKING_TOKENS=0`、`--thinking disabled`、およびモデルごとの thinking トグルが、**デフォルトで思考（thinking）するモデルの思考を実際にオフにします**（Claude API 経由が対象、サードパーティプロバイダは変更なし）。推論をほとんど必要としない**大量の機械的なパス**（定型変換、単純な置換、フォーマット整理などのサブエージェント作業）では、思考トークンがコストとレイテンシだけを増やしがちです。そこで thinking をオフにすると、1回あたりのコストと応答時間を削減できます。

```bash
# このセッションで thinking を無効化（デフォルトで思考するモデルも思考しない）
export MAX_THINKING_TOKENS=0
claude
```

逆に、アーキテクチャの判断や難度の高いデバッグなど、推論が結果を左右する作業では thinking をオンにしておくべきです。「どこに推論の予算を使うか」を作業の性質に応じて切り分けることは、6/15 の Programmatic Usage Credits 開始を前にしたコストガバナンスの一軸です。

[GitHub v2.1.166](https://github.com/anthropics/claude-code/releases/tag/v2.1.166)

### v2.1.166 — 中継されたクロスセッションメッセージがユーザー権限を持たなくなった（6/6）

v2.1.166 で、**セッション間メッセージ中継（cross-session messaging）のハードニング**が入りました。他の Claude セッションから `SendMessage` で中継されたメッセージは、**もはやユーザー権限（user authority）を持ちません** — 受信側は中継された権限要求を拒否し、auto モードはこれをブロックします。あるセッションが別のセッションをツールのように操って権限を回避する経路を塞ぐ変更です。あわせて入った権限関連の修正も点検する価値があります。

- **不正な項目が1つあると残りのポリシーまで無効化されていた問題を修正**：managed settings に無効な項目が1つあると、そのファイルの**残りの有効なポリシーの適用まで静かに無効化**されていた問題が修正されました。
- **`${VAR}` 使用時に MCP サーバーのポリシーがマッチしない問題を修正**：managed-settings の `allowedMcpServers`/`deniedMcpServers` が `${VAR}` 参照を使うとマッチしなかった問題が修正されました。

[GitHub v2.1.166](https://github.com/anthropics/claude-code/releases/tag/v2.1.166)

## おすすめコラム＆読み物

- **「AI ネイティブなスタートアップの作り方」**：競争優位は、より良いモデルではなく**規律（discipline）** から生まれるという6ステップのフレームワークです。ワークフローを体系的にマッピングし、コンテキストシステムを構築し、**評価（eval）とともにスキルをコード化**し、週次の改善ループを回せば、小さなチームが従来型の組織より指数関数的に速く学習する、と説きます。「スキルをコード化せよ」という点で、上の Anthropic のスキル記事とまったく同じ筋の実践論です。[cyberfund（X）](https://x.com/cyberfund/status/2058950286324986294)

- **「AI が自らを作るとき：再帰的自己改善に向けた我々の進捗」 — Anthropic**：Claude の**タスク完遂可能な時間（task horizon）が約4か月ごとに2倍**になっており、この傾向と十分な計算資源が組み合わされば、AI がいずれ後継モデルを自ら設計・構築する可能性がある、という分析です。ただし「どの問題が最も重要か」という**研究上の判断は依然として人間の比較優位**に残り、意味のある安全策には複数のフロンティアラボ間の**検証可能な国際協調**が必要だと指摘します。エージェントにますます多くを委ねる我々が、その軌跡をどう捉えるべきかを問い直す一篇です。[Anthropic](https://www.anthropic.com/institute/recursive-self-improvement)

- **「Claude は rsync のバグを増やしたか？」**：Claude の支援を受けた rsync のリリース（v3.4.2・v3.4.3）が、過去のリリースより本当にバグが多かったのかを**データで**検証した記事です。コミット10件あたりの重大度加重バグ数を指標に、順列検定・Fisher の正確確率検定で **p 値 46%・74% — 「Claude のリリースが異常にバグが多いという統計的証拠はない」** と結論づけます。むしろ Claude 以前の v3.4.1 がデータセット全体で最もバグが多いものでした。AI コーディングへの印象論的な批判を**数字で反論する**、落ち着いた分析です。[alexispurslane.github.io](https://alexispurslane.github.io/rsync-analysis/)

## 注目プロジェクト＆ツール

- **harness-starter-kit — コーディングエージェントが同じ失敗を繰り返さないためのレポ「ハーネス」**：`AGENTS.md`・決定ログ（decision log）・失敗追跡ファイルで、**プロジェクトのルールと決定履歴をレポ内に永続化**し、エージェントがセッションをまたいで過去の文脈を参照できるようにするオープンソースのスターターキットです。Django・Next.js・FastAPI 向けのワークフロースクリプトとプロファイル例を提供します。ルールを「強制」していた oh-my-harness とは異なり、こちらは**決定と失敗の記憶を蓄積する**点に重きがあります。[GitHub](https://github.com/baskduf/harness-starter-kit)

- **エージェントメモリアーキテクチャプラグイン — メモリのノイズと decay 問題に対処**：エージェントを「プロンプト」ではなく**実際のファイルを持つレポ**として構造化し、キュレーター（curator）が監督する**チケットベースのメモリ**でノイズと記憶の減衰（decay）に対処するオープンソースプラグイン（Apache 2.0）です。Claude・Codex・Gemini のランタイム間で**ベンダーロックインなしに移植**でき、自己進化する単一エージェント、検証層を備えたオーケストレーション型マルチエージェント、既存ワークスペースの再パッケージ化という3つのモードを提供します。長時間セッションの「記憶」問題に正面から取り組む試みです。[GitHub](https://github.com/jeongmk522-netizen/agent_agentlas_core_engine_meta_agent)
